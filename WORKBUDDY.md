# WORKBUDDY.md —— 入境游定制协作工作台 · 项目级全局规则

> 本文件是 AI 开发的**唯一全局约束入口**。所有模块细节拆进 `doc/`，本文档只立规矩与索引，避免上下文爆炸。

## 1. 开发原则

- **KISS**：优先简单方案，复杂设计必须说明理由。
- **第一性原理**：从协作本质（多方报价对齐）出发，不堆技术栈。
- **不确定就追问**：需求模糊、边界不清、数据缺失必须停下提问，不许猜。
- **一次只解决一层**：PRD 定稿 → 架构批准 → 再编码；不跨层混做。
- **结果可验证**：每阶段都有明确验收信号（见 `doc/05-验收标准.md`）。

## 2. 项目专属约束

### 2.1 三方角色模型
- **一手 PandaKing（地接社总部）**：系统 owner，可见全部成本①②，负责确认/采纳/出游客报价。
- **境外旅行社**：可发起结构化「行程规划草案」并请求一手确认；可加价生成游客报价；可填写客户档案。
- **省地接社**：接收成本询价 H5，填写成本价①②，隔离于其他省地接社。

### 2.2 利润与成本
- **模式 A+B 并存**：数据层成本①（省地接社成本）与成本②（一手利润/加价）**强制分离存储**。
- 输入交互可选：成本+利润分解录入，或直接含利润报价。
- 游客报价 = 成本① + 成本② + 旅行社加价。

### 2.3 多语言
- 界面与 H5 支持 **EN / ZH / TH / RU** 四语，字段级 i18n，缺失翻译回退 ZH。

### 2.4 微信协作
- 微信是主沟通通道，系统提供「结构化引擎」：每个路线/每次修改/每份报价有专属 H5 链接（带版本号），复制粘贴到微信。
- 双向协作闭环：旅行社发起草案 → 一手确认/反馈 → 循环。

### 2.5 脱敏合规（个人信息保护法）
- 案例展示仅可公开：目的地/天数/主题/亮点/参考价格区间。
- **禁止公开**：客户真名、证件号、精确出行日期、客户专属合同价。
- 案例必须由「已确认」路线派生，发布前强制脱敏校验 + 发布人二次确认。

### 2.6 技术栈（架构 v1，已批准）
- 前端：**Vue 3 + Vite + TypeScript**
- 后端：**Node.js + TypeScript + NestJS**
- 数据库：**PostgreSQL + JSONB**（Prisma ORM）
- 鉴权：微信 OAuth2 + 邀请制账号
- 部署：**CloudBase 云托管（Docker）+ EdgeOne CDN**
- 仓库形态：单仓 monorepo（`frontend/` + `backend/` + `doc/` + `design/` + `prototype/`）

## 3. 代码规范

- 变量/函数：英文，语义清晰；组件 PascalCase，文件 kebab-case。
- 注释：**中文**，解释「为什么」而非「做什么」。
- 函数单一职责，参数 ≤ 5 个。
- 提交：小步提交，commit message 用**中文**，格式「类型：描述」（feat/fix/docs/refactor/chore）。
- 前端状态用 Pinia；后端分层 controller/service/repository。

## 4. 安全红线

- 密钥、密码、Token、微信 AppSecret 必须放环境变量（`.env`），**禁止写进代码或文档**。
- 不执行 `rm -rf`、数据库 drop/truncate 等危险操作，除非用户二次确认。
- 对外发布内容（案例、H5）需用户确认；脱敏校验失败禁止发布。
- 省地接社数据严格隔离，跨社查询必须拦在 service 层。

## 5. 沟通规则

- 回答用**中文**。
- 先给结论，再给细节。
- 遇到选择时，提供**推荐选项并说明理由**。
- 重大架构/范围变更，先停闸门请示，不擅自推进。
