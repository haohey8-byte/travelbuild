## 1. 背景与目标 (Why)

### 1.1 业务背景

你的核心业务是**包车带司机**的入境游定制服务。行程天数直接决定包车天数，酒店数量直接决定房费，景点直接决定门票——**行程和报价本质是一件事的两面**。你在整条链路中处于**枢纽位置**，一对多对接上下游：

```
多个境外旅行社          一手地接社（PandaKing）        多个省地接社
```

> **命名约定** ★（v0.9.1 新增）：

**四方角色定义** ★（v0.9 更清晰定义）：

| 角色 | 职责 | 数量 | 关系 |
|---|---|---|---|
| **境外游客** | 提需求（目的地/天数/预算/偏好），反馈修改意见 | 多人 | 不直接对接，需求经旅行社转达 |
| **境外旅行社路线规划人员** | 对接游客、整理需求、转达给一手地接社、**在系统内参与行程+报价反馈**、在旅行社报价基础上**加利润生成对游客报价** | 多家×多人/家 | 上游客户，你给他们报价 |

**关键数量关系**：
- 1个 PandaKing（一手地接社） ↔ N个境外旅行社（如泰国A旅行社等） ↔ M个境外游客（每个旅行社带多个游客）
- 1个 PandaKing ↔ K个省地接社（按目的地分配，如新疆A旅行社、四川社、云南社等）
- 同一时间段内在跟进的行程可能有 **5-15条**，每条都可能在不同版本阶段

> **四个关键认知** ★（v0.9.1 更新）：
> 1. 一手地接社与省地接社是**协作关系**。**常规/成熟行程**可由一手地接社自行规划和报价，提高效率；**定制/新目的地行程**由省地接社提供行程规划和成本报价，一手地接社加利润后对外报价。**推荐"自行优先，按需转协作"**：先自行出初版抢占客户，需要时再引入省地接社细化。★（v0.9 新增：省地接社可通过 H5 接收成本询价并填写成本价，不再仅靠微信口头沟通）★（v0.9.1 新增：省地接社不仅填写成本价，还可在 H5 内编辑分配给自己的行程内容、反馈行程规划修改建议，成为行程规划的积极参与者）
> 2. **微信仍是主沟通通道**，系统不替代微信，而是给微信装上"结构化引擎"——每个客户、每次行程修改、每份报价都有专属 H5 链接，带标注粘贴到微信，让微信群里的沟通有归属、有版本、可回溯。★（v0.9 强化：行程报价合并后"保存并通知"一条 H5 链接含行程+报价；省地接社侧也通过 H5 闭环完成成本询价）★（v0.9.10 升级：境外旅行社销售可直接发起结构化「行程规划草案」（按天行程框架+预算区间）并提交给一手地接社，一手可确认采用或回传修改反馈，协作方向双向化）
> 3. **沟通分双通道**：简单事项在微信直接聊（不强制进系统，可选手动备注留痕）；复杂事项（行程修改、报价确认、需求变更、成本询价、行程规划反馈）通过 H5 系统交互确认并记录沉淀，有价值的交互沉淀为知识库。
> 4. ★（v0.9 新增）**行程和报价是一件事的两面**：包车带司机是核心业务，行程天数决定包车天数，酒店数量决定房费，景点决定门票。行程修改必然影响报价，因此 v0.9 将行程和报价合并为「行程报价单」单一模块，修改联动不遗漏。
> 5. ★（v0.9.1 新增）**账号管理简化为邀请制**：管理者邀请→员工微信授权→直接获得权限，取消审核流程，首次打开H5完成微信授权即获得权限，降低使用门槛。
> 6. ★（v0.9.1 新增）**境外旅行社和省地接社信息隔绝**：泰国A旅行社（境外旅行社）和新疆A旅行社（省地接社）在系统信息流转中被隔离，互相不可见。所有信息经一手地接社（PandaKing）中转，旅行社不可指定省地接社，省地接社由一手地接社分发后关联。

### 1.2 现状问题

前期所有沟通都在**微信群**内完成，痛点集中在三个层面：

**层面一：信息归属与版本管理**

| 痛点 | 表现 | 业务影响 |
|---|---|---|
| P1 多游客多路线交叉混乱 | 5-15条路线同时在跟进，A游客的反馈回给了B旅行社，C旅行社的报价发给了D游客 | 商业隐私泄露、客户流失 |
| P2 版本管理失控 | v1/v2/v3在微信里来回发，旅行社说"按上次发的"，你不知道是哪个"上次" | 用旧版报价→亏利润；用新版但旅行社看旧版→客户混乱 |
| P3 换人交接断档 ★（v0.9 补充） | 负责某条路线的员工离职/休假，接手的人面对一堆微信聊天记录，完全不知道上下文 | 交接成本极高，可能丢单 |
| P7 历史版本丢失 | 微信聊天记录被清理或换手机，之前的修改过程完全消失 | 无法追溯、扯皮无据 |

**层面二：协作信息流转效率**

| 痛点 | 表现 | 业务影响 |
|---|---|---|
| 沟通记录无沉淀 | 每次修改只在微信里说，没有结构化记录 | 换人接手就断档（P3） |
| 重复问题反复答 | 游客常问的问题每次重新打字 | 回复慢，游客体验差 |
| 价格口径混乱 | 成本价、对旅行社报价、对游客报价混在一起说 | 报错价、泄底价（P4） |
| P5 变更后成本难核算 ★（v0.9 补充） | 改了3次酒店、换了车型、加了体验项目，最后利润到底多少全靠脑子算；**行程和报价分属两个模块，修改行程不会自动联动报价** | 利润被蚕食、报价可能亏本 |
| P9 反馈口语化难结构化 ★（v0.9 补充） | 游客说"那里不好不要了加个别的"，你理解偏差；**行程和报价的反馈是分开的，游客反馈往往同时涉及行程和价格（"换酒店+多少钱"），分开反馈割裂** | 改错方向、浪费来回 |
| 行程反馈来回低效 | 旅行社反馈修改意见靠微信口头描述，一手地接社改完再口头通知，来回多轮说不清 | 沟通效率低，容易遗漏 |
| P11 微信通知易遗漏 ★（v0.9 补充） | 同时跟进多条路线，某条的修改通知淹没在消息列表里；**多条路线同时跟进时缺乏优先级排序** | 遗漏修改、客户等急 |

**层面三：多账号协作混乱**

| 痛点 | 表现 | 业务影响 |
|---|---|---|
| 多旅行社交叉 | 旅行社 A 的客户信息发给了旅行社 B | 商业隐私泄露 |
| P6 多省地接社报价交叉 ★（v0.9 补充） | 四川社和云南社的成本报价在同一个微信聊天里，互相看到了对方的价格；**省地接社侧没有 H5 协作闭环，成本询价仍走微信** | 供应商比价、被动 |
| 分配不明确 | 不清楚某个客户行程该找哪个省地接社 | 踢皮球，延误 |
| 旅行社加价不透明 | 旅行社在系统外加价，改了多少、报给游客多少一手地接社不知道 | 无法掌握最终成交价，难以定价策略调整 |

### 1.3 产品目标

把"微信群里的口头协作"变成"微信 + 结构化引擎"的协作模式：
- **微信仍是主沟通通道**，但每个客户、每次修改、每份报价都有**专属 H5 链接**，带标注粘贴到微信，一眼识别归属；
- **行程报价合并为单一页面** ★（v0.9）：行程区块（按天编排）和报价区块（按项按车型分列）在同一页面，行程修改后报价区块自动标记"需更新"，报价行跟随行程联动（如 Day3 换酒店→房费行自动标红待改），不再分两个模块手动同步；
- **行程报价"保存并通知"** ★（v0.9）：保存行程报价单后一键生成**一条 H5 链接含行程+报价**，复制粘贴到微信通知旅行社，旅行社在微信内打开 H5 即可查看行程+报价并反馈修改建议；
- **省地接社 H5 协作闭环** ★（v0.9）：一手地接社向省地接社发"成本询价 H5 链接"，省地接社在微信内打开 H5 查看行程需求+填写成本价→保存并通知→一手地接社收到，成本询价不再散落微信；
- **路线管理（看板视图 + 列表视图）** ★（v0.9 新增 → v0.9.8 合并）：原独立的「多路线看板」与「路线工作台」合并为单一模块「路线管理」，顶部以「看板视图 / 列表视图」切换呈现——看板视图保留卡片式全局概览（一屏看全部路线/状态/待办、超时标红），列表视图提供高密度筛选/排序/多选/批量删。二者共享同一详情视图（Tab1 行程报价 + Tab2 客户信息），作为 MVP 核心功能；
- **微信驱动三方协作闭环** ★（v0.9）：闭环A（你↔旅行社：行程+报价协作）+ 闭环B（你↔省地接社：成本询价协作），全程微信驱动；
- **H5反馈同时覆盖行程和报价** ★（v0.9）：反馈可关联到具体行程天，也可关联到具体报价行，如"D3酒店太贵，建议换¥800以内的"——同时标注行程 Day3 和报价酒店行；
- **双通道沟通**：简单事项微信直接聊，复杂事项 H5 系统交互确认并记录沉淀；
- 每一次修改都有版本 + 备注 + 修改人记录；**行程调整后系统自动生成建议备注，供用户勾选确认**；
- 每一次报价都有快照，且**成本价 / 对旅行社报价 / 对游客报价分层隔离**；
- **5 级报价链路全程系统内闭环**：成本价 → 一手加利润 → 旅行社报价 → 旅行社加价 → 游客报价；
- **包车分车型报价**：同一行程按 7座/9座/14座等不同车型生成不同报价方案；
- **常规行程可自行规划报价**，不强制依赖省地接社，提高出单效率；
- 常见问题形成知识库，**系统自动提示历史答案**，帮助快速回复游客；
- **多个境外旅行社、多个省地接社账号体系**，数据严格隔离，**账号邀请权限分层下放**；
- **行程/报价 PDF 支持多语言导出**，**行程内容默认中文填写，AI 翻译引擎自动翻译目标语言**。

### 2.3 技术栈与部署形态（来自架构设计 v1）★（v0.9.11 新增）

| 维度 | 决策 | 说明 |
|---|---|---|
| 前端框架 | **Vue 3 + Vite** | H5 组件库按微信 H5 体验选型（候选 Vant / NutUI）；由现有原生 HTML/CSS/JS 原型渐进迁移 |
| 后端 | **Node.js（TypeScript）+ NestJS** | 模块化支撑权限/路由边界；REST/JSON API |
| 数据库 | **PostgreSQL + JSONB** | 关系数据走表，行程/报价等结构化内容走 JSONB 列，兼顾灵活与强一致 |
| 鉴权 | **微信网页授权（OAuth2）+ 邀请制** | 与 3.x 一致：管理者邀请 → 微信授权 → 直接获权；MVP 内可先以"邀请链接 + openid 绑定"模拟，正式环境接 OAuth2 |
| 多语言 | **i18n（EN/ZH/TH/RU）** | 与 Q6.1–Q6.3 一致；保存并通知时生成译文快照 |
| 利润模式 | **模式 A + B 并存** | 数据层强制分离成本价①/利润②（见 4.2.18）；输入交互可选成本+利润分解或含利润直接报价 |
| 部署 | **腾讯云 CloudBase 云托管（Docker）+ 静态资源 CDN/EdgeOne** | Vue 静态产物 → 对象存储 + CDN/EdgeOne；Node API → Docker 容器（CloudBase 云托管或 CVM）；PostgreSQL → 腾讯云 PostgreSQL（或 CloudBase 云数据库 PostgreSQL）。**不再依赖 CloudBase 文档数据库** |
| 环境变量 | 密钥管理（不入库） | `DATABASE_URL` / `WX_APPID` / `WX_SECRET` / `JWT_SECRET` / `H5_BASE_URL` |

**与 PRD 原假设的偏差（本版已回写）**：
1. **MVP 范围上调**：案例展示由"Phase 2 / Later"拉回 MVP 一期（见 2.1、4.8、6.2、6.3）。
2. **部署形态变更**：由"CloudBase 文档库"改为"CloudBase 云托管 Docker + CDN"，后端为自建 Node+PostgreSQL 应用。

### 3.2 核心用户故事

#### 故事 1 - 创建客户档案
> **作为** 境外旅行社销售
> **我希望** 在新客户首次咨询时立即为他创建一个客户档案
> **以便** 后续所有的沟通和方案都自动归属到这个客户，不会再"回复错人"。

#### 故事 2 - 复制模板生成 v1
> **作为** 一手地接社运营
> **我希望** 从线路模板库一键复制，并按当前客户需求修改
> **以便** 快速出 v1 行程骨架。

#### 故事 3 - 自动版本管理 + 变更备注智能生成
> **作为** 一手地接社运营
> **我希望** 每次修改行程报价单后系统自动保存为新版本，且系统**自动检测变更内容并生成建议备注供我勾选确认**，我也可以手工补充备注
> **以便** 不再混淆最新版本，且变更备注不用每次从零写，省时又准确。

#### 故事 4 - 5级报价 + 价格隔离
> **作为** 一手地接社运营
> **我希望** 省地接社提交成本价后（或我自行填写成本价后），在上面加利润生成对旅行社报价，旅行社**看不到成本价**；旅行社在旅行社报价基础上加价生成对游客报价
> **以便** 保护我的利润空间，同时掌握旅行社加价和最终游客报价。

#### 故事 5 - 生成带标注的 H5 链接粘贴到微信
> **作为** 一手地接社运营
> **我希望** 行程报价修改后，系统自动生成一条带标注的 H5 链接（**含行程+报价** ★），我可以一键复制粘贴到微信群
> **以便** 微信群里的沟通有清晰归属。

#### 故事 6 - 链接微信登录与行为追踪
> **作为** 一手地接社管理者
> **我希望** 发给旅行社/省地接社的链接通过微信授权登录后（需通过邀请获得权限），对方才能查看数据，且对方打开链接的时间和行为被记录
> **以便** 知道对方是否看了行程报价，同时防止信息泄露。

## 4. 功能详细说明

### 4.1 路线工作台（客户档案 + 行程报价单合并）(M1+M2) ★（v0.9.4 合并）

> **v0.9.4 重大变更**：v0.9.3 中客户档案（M1）和行程报价单（M2）是两个独立模块，各有独立菜单、列表、搜索。v0.9.4 合并为「路线工作台」单一模块——客户档案内容作为行程报价单详情页的 Tab2「客户信息」展示。合并理由：客户档案与路线数据 1:1 完全重叠（同一条记录被两个模块各展示一遍），列表重复、搜索重复、跳转多余。合并后菜单从 6 项减为 5 项，详情页通过 Tab 切换在「行程报价」和「客户信息」间零跳转切换。

#### 4.2.1 设计目标

#### 4.2.2 行程规划双模式

创建客户行程报价单时，一手地接社运营选择行程模式：

| 模式 | 适用场景 | 流程 | 成本价填写方 |
|---|---|---|---|
| **自行模式** | 常规/成熟线路 | 选模板 → 自行修改行程 → 自行填写成本价 → 加利润 → 生成报价 | 一手地接社 |
| **协作模式** ★（v0.9 更新） | 定制/新目的地线路 | 选模板/创建空白 → 分配省地接社 → **发送成本询价H5链接** ★ → 省地接社在H5内填写成本价 → **省地接社"保存并通知"通知一手地接社** ★ → 一手加利润 → 生成报价 | 省地接社（通过H5） |

> **"自行优先，按需转协作"主流程**：
> - 推荐路径：一手地接社先用自行模式快速出初版行程+报价 → 后续如需当地资源时切换为协作模式
> - 切换无限制：自行 ↔ 协作可随时互转
> - 切换时自动记录到沟通时间线
> - 切换不影响已有报价快照
> - 自行模式成本价参考机制：填写成本价时系统自动展示同目的地/同类资源的历史报价参考
> - ★（v0.9 新增）协作模式下，一手地接社在行程报价单页面点击"发送成本询价"→生成成本询价H5链接→复制到剪贴板→粘贴到微信发给省地接社（详见 4.4）

> **双向协作入口（v0.9.10 新增）：境外旅行社可直接发起行程规划草案**
> - 除一手主导规划外，境外旅行社销售也可在创建客户档案后**直接发起行程规划草案**（按天结构化行程框架 + 预算区间，见 4.1.1「行程规划草案」字段与故事36）
> - 旅行社点击"提交给一手地接社"→ 路线进入「待一手确认」→ 一手地接社**确认采用**（草案转正为「行程报价单」、归属一手，补成本/利润出报价）或**回传修改反馈**（在草案副本上修改并标注，路线转「待旅行社修订」）
> - 旅行社修订后再次提交，循环直到一手确认；草案与行程报价单共用同一套行程数据结构，仅在「发起方/确认方」角色与归属上区分
> - 设计原则（KISS）：不新增独立模块与数据表，草案复用行程报价单数据结构 + 客户状态机 + 看板状态标签，仅在「提交方」与「状态」维度扩展

#### 4.2.4 行程修改→报价联动机制 ★（v0.9 新增）

**这是 v0.9 合并版的核心机制**——行程修改后报价区块自动联动标记，防止"改了行程忘记改报价"。

**联动规则**：

| 行程变更类型 | 联动的报价行 | 联动动作 |
|---|---|---|
| 修改某天酒店 | 对应天的酒店行 | 标记"需更新"（标红） |
| 新增一天 | 所有按天计费行（酒店/包车/门票等） | 标记"需更新"+提示"新增Day N，请补充报价" |
| 删除一天 | 所有按天计费行 | 标记"需更新"+提示"已删除Day N，请核减报价" |
| 修改某天景点 | 门票行 | 标记"需更新" |
| 修改某天用餐 | 餐饮行（如有） | 标记"需更新" |
| 修改交通方式 | 包车行 | 标记"需更新" |
| 修改出行人数 | 车型方案行 | 标记"需更新"+提示"出行人数变更，请核实车型推荐" |

**联动展示**：

```
┌──────────────────────────────────────────────────────────┐
│  💰 报价区块                              [⚠ 2项需更新] 🔴│
│                                                          │
│  项目       │ 7座方案  │ 9座方案  │ 14座方案 │ 备注     │
│  ├─────────────────────────────────────────────────────┤│
│  │酒店(5晚)  │ ¥6,000  │ ¥6,000  │ ¥6,000  │五星级    │
│  │包车(5天)  │ ¥4,000  │ ¥6,000  │ ¥9,000  │含司机    │
│  │⚠ 门票     │ ¥3,000  │ ¥3,000  │ ¥3,000  │Day5景点变│
│  │⚠ 服务费   │ ¥2,000  │ ¥2,000  │ ¥2,000  │天数变更  │
│  └─────────────────────────────────────────────────────┘│
│  ↑ 标红的行需要您更新报价，行程已变更但报价未同步          │
└──────────────────────────────────────────────────────────┘
```

**操作流程**：
1. 用户在行程区块修改某天内容（如 Day3 换酒店）
2. 系统自动检测变更类型，将关联的报价行标记为"需更新"（标红高亮）
3. 报价区块顶部显示"⚠ N项需更新"汇总提示
4. 用户点击标红的报价行，更新价格
5. 更新后该行标记变为"已更新"（标绿，后恢复正常）
6. 所有"需更新"项处理完毕后，汇总提示消失
7. 如用户在仍有"需更新"项时点击"保存并通知"，系统弹出确认："仍有2项报价未更新（门票、服务费），确认保存并通知？"——用户可选择"继续保存"或"先去更新"

> **设计目标**：减少因行程变更导致报价遗漏的风险，保证行程和报价一致性。联动是"提醒"而非"自动改价"——系统不自动修改价格数字（因为价格需要人工确认），只标记哪些行需要关注。

#### 4.2.5 变更备注智能生成

行程报价单调整后保存新版本时，系统**自动检测变更内容**并生成建议备注：

**检测逻辑**：
- 对比当前版本与上一版本的差异（行程区块+报价区块均检测）★（v0.9 扩展：报价变更也纳入检测）
- 识别变更类型（新增/删除/修改 Day、酒店/餐饮/交通/景点变更、天数变化、报价行价格变更、车型方案变更等）★
- 自动生成结构化建议备注

**建议备注示例**：
```
系统检测到以下变更：
☑ D3 酒店：九寨沟假日酒店 → 九寨沟希尔顿（价格 +800/晚）
☑ D5 景点：删除购物点，改为自由活动
☑ 门票报价行：¥3,000 → ¥2,500（Day5景点变更联动更新）★
☐ D7 用餐：团餐 → 自助餐
☐ 新增 D8：返程日
☐ 包车报价行：7座方案 ¥4,000 → ¥4,800（天数增加联动更新）★

请在上方勾选需要记录的变更，也可在下方手工补充：
┌─────────────────────────────────────────────┐
│ （手工输入框，可补充系统未检测到的备注）         │
└─────────────────────────────────────────────┘
```

**操作方式**：
1. 系统自动勾选检测到的变更项（用户可取消勾选）
2. 用户可手工补充额外备注
3. 至少需要保留一项备注（勾选项或手工输入），否则不允许保存
4. 最终备注 = 勾选项（系统生成）+ 手工补充（用户输入），合并存储

> **设计目标**：减少用户每次手写备注的负担，同时保证变更记录的准确性和结构化。★（v0.9 扩展：行程变更和报价变更统一检测，联动更新的报价行也自动出现在建议备注中）

| 价格层级 | 谁填写 | 谁可见 | 说明 |
|---|---|---|---|

> - 旅行社看不到 ① 成本价 和 ② 一手利润
> - 一手地接社看得到 ④ 旅行社加价 和 ⑤ 游客报价，但**不可修改**

> - ★（v0.9.1 强化）**省地接社全部不可见的价格**：② 一手地接社利润、③ 对旅行社报价、④ 旅行社加价、⑤ 对游客报价——全部不可见
> - ★（v0.9.1 强化）**省地接社之间的成本价互相不可见**：新疆A旅行社看不到四川社的成本价
> - ★（v0.9.1 强化）**境外旅行社看不到省地接社的任何信息**：成本价、省地接社名称、行程编辑和反馈——全部不可见
> - 游客只看到 ⑤ 最终报价（通过旅行社转发的 PDF/H5）

#### 4.2.10 价格计算逻辑

- **自行模式**：一手地接社自行填写成本价（含分车型）
- **协作模式** ★（v0.9 更新）：成本价由省地接社通过 H5 提交（含分车型），一手地接社收到后在行程报价单内确认使用
- **一手利润**由一手地接社设置（百分比或金额二选一）
- **对旅行社报价** = 成本小计 + 一手利润（系统自动计算，按车型分别计算）
- **旅行社加价**由旅行社销售设置（百分比或金额二选一）
- **对游客报价** = 对旅行社报价 + 旅行社加价（系统自动计算）
- 一旦点击"生成报价快照"，所有数据冻结
- 后续如需改价：必须创建新版本 + 新报价快照
- ★（v0.9 新增）行程修改后报价行联动标记"需更新"，用户更新后触发版本+1
- ★（v0.9 新增）改行程或改报价都触发共享版本号+1

#### 4.2.12 行为规则

- 每次"保存为新版本"必须确认变更备注（**系统建议+手工补充混合模式**），变更检测范围含行程+报价 ★

- 每次版本变更自动生成 H5 链接 + 自动在沟通记录中心生成记录
- 自行模式下，一手地接社运营直接填写成本价
- 协作模式下，省地接社通过 H5 填写成本价，一手地接社收到后确认使用 ★（v0.9）
- **"保存并通知"生成的 H5 链接包含行程+报价协作功能** ★（v0.9 合并）：旅行社打开后可查看行程+报价详情+提交反馈修改意见（可同时关联行程天和报价行）；一手地接社打开后可查看反馈+直接修改行程和报价
- **"保存并通知"按协作上下文选链，避免误发** ★（v0.9.12 修正）：路线**未关联省地接社**→生成对客只读「行程报价协作链接」发境外旅行社；路线**已关联省地接社**→改生成「省地接社可编辑协作链接」发省地接社。该链接由后端幂等端点 `ensureProvincialShare` 生成，同一路线+同一省地接社始终复用唯一链接，重复点「保存并通知」不会新增链接。对客只读链接若误发给省地接社，省地接社会看到不可编辑、空白的页面，须用省地接社可编辑链接

#### 4.2.15 选择与删除操作 ★（v0.9.5 新增）

路线工作台此前仅支持"编辑 + 新建"，缺少**删除**与**批量选择**能力。v0.9.5 补充如下操作，覆盖"增 / 改 / 删 / 选"四类基础 CRUD 能力：

**（一）列表视图 — 多选与批量删除**

| 操作 | 触发方式 | 行为 |
|---|---|---|
| 单条勾选 | 列表卡片左上角勾选框 | 选中/取消该条路线；卡片进入"已选中"高亮态（品牌色边框 + 浅蓝底） |
| 全选 | 工具栏「☐ 全选」 | 选中/取消当前筛选结果内的全部路线（受搜索、状态筛选影响） |
| 批量操作栏 | 只要有 1 条被选中即出现 | 顶部浮层显示「已选 N 项」+「🗑️ 批量删除」+「取消选择」，跟随列表滚动 |
| 单条删除 | 卡片右上角 hover 显示 🗑️ | 对该条路线单独删除（同样走二次确认） |

**批量删除交互**：
1. 勾选若干路线 → 出现批量操作栏
2. 点击「批量删除」→ 弹出二次确认弹窗，列出将被删除的路线名称清单
3. 确认 → 从路线列表与关联客户档案中一并移除 → Toast 提示「已删除 N 条路线」→ 清空选择态并刷新列表

**选择态规则**：
- 选择是**视图态**（不写库、不锁定），仅用于批量操作；刷新页面后清空
- 全选范围 = 当前筛选+搜索后的结果集，跨页/跨筛选不会误选未展示项
- 删除某条后，若其正处于详情页，自动退回列表视图

**（二）客户信息 Tab — 单条删除**

| 操作 | 触发方式 | 行为 |
|---|---|---|
| 删除客户 | Tab2「客户信息」头部「🗑️ 删除」按钮（与「✏️ 编辑」并列） | 弹出二次确认 → 确认后从路线+客户档案一并删除 → 退回列表视图 |

**数据一致性约束**（关键）：
- 路线（routes）与客户档案（customers）是 **1:1 绑定**的同一业务主体
- 任何删除操作都同时作用于 `routes` 与 `customers` 两条记录，保证**不产生孤立数据**（不会出现"有路线没客户"或"有客户没路线"）
- 删除为**软硬均不可逆**操作，必须二次确认，确认弹窗明确列出受影响对象，杜绝误删

#### 4.2.16 行程天与报价项的增删操作 ★（v0.9.6 新增）

v0.9.5 前原型中，详情页 Tab1「行程报价」存在两处 CRUD 不完整：

1. **行程区块**：有编辑(✏️)和添加(+ 添加一天)，**缺删除天**
2. **报价区块**：有编辑(行内数字输入)，**缺新增报价项和删除报价行**

本次补齐为完整 CRUD 能力。

**（一）行程天 CRUD**

| 操作 | 触发方式 | 交互行为 | 后端效果 |
|---|---|---|---|
| 新增天 | 行程区块 header「＋ 添加一天」或底部「＋ 添加一天（Day N+1）」 | 即时在列表末尾插入新 Day，默认城市"待定"、景点"待规划"、用餐/交通"待定"；Toast 提示 | `itinerary.days` push 新对象；所有 `quote.items[i].needsUpdate = true` |
| 编辑天 | 每天卡片右侧 ✏️ 按钮 → 或直接点击 contenteditable 字段 | 进入行内编辑模式（contenteditable），修改住宿/景点/用餐/交通任一字段 | 字段值即时写入 DOM（前端暂存）；保存版本时一并提交 |
| 删除天 | 每天卡片右侧 🗑️ 按钮 | 二次确认弹窗 → 显示将删除的天号、城市、景点；提示"后续天数将自动重新编号" | 确认后从数组 splice 该天 → 遍历剩余天数重新编号 `.day = index + 1` → 所有报价项标记 needsUpdate=true |

**行程天删除约束**：
- **至少保留 1 天**：当 `days.length <= 1` 时禁用/隐藏删除按钮，Toast 提示"行程至少需要保留1天"
- **自动重编号**：删除 Day3 后，原 Day4→Day3、Day5→Day4… 保证 day 序号连续无断
- **联动标记**：无论增/删/改行程天，均触发报价区块全量标记 `needsUpdate = true`

**（二）报价项 CRUD**

| 操作 | 触发方式 | 交互行为 | 后端效果 |
|---|---|---|---|
| 新增报价项 | 报价区块 header「＋ 新增报价项」（primary 蓝色按钮） | 弹出 Modal 表单：（必填）项目名称 +（可选）备注 + 三车型成本价（7座/9座/14座数字输入）→ 自动生成 key=`custom_时间戳` | `quote.items` push 新对象；渲染刷新后在表格末尾显示新行 |

| 删除报价项 | 报价表每行项目名旁 🗑️ 按钮（hover 显示） | 二次确认弹窗 → 显示将被删除的项目名 → 提示"5级价格合计将自动重新计算" | 从 `quote.items` splice 移除 → 重新调用 calcQuote() → 价格层级区域刷新 |

**报价项约束**：
- **新增表单校验**：项目名为必填项，空提交时 Toast 警告"请填写项目名称"

- **删除后重算**：删除报价项后立即重算 costSubtotal → profit → agencyQuote → markup → touristQuote 全链路

### 4.4 H5 链接标注分享与微信协作闭环 (M3) ★（v0.9 大幅扩展）

#### 4.4.1 设计目标

微信是主沟通通道，系统不替代微信。系统为每个关键对象生成带标注的专属 H5 链接，用户一键复制粘贴到微信群，对方点开即可在 H5 页面查看详情、操作、备注。

★（v0.9 核心扩展）：**微信驱动三方协作闭环**。H5 页面是协作面，微信是通知通道。
- **闭环A（你 ↔ 旅行社）**：行程+报价协作——一手地接社"保存并通知"生成含行程+报价的H5链接→微信粘贴→旅行社微信内打开H5→查看行程+报价+反馈（可同时标注行程天和报价行）→复制链接通知→一手地接社微信内H5修改→保存并通知→循环
- **闭环B（你 ↔ 省地接社）** ★（v0.9 新增）：成本询价协作——一手地接社"发送成本询价"生成成本询价H5链接→微信粘贴→省地接社微信内打开H5→查看行程需求+填写成本价→"保存并通知"→一手地接社收到成本报价→在行程报价单内加利润→生成对旅行社报价

#### 4.4.2 链接类型

| 链接类型 | 触发时机 | 链接内容 | 通知方式 |
|---|---|---|---|
| 客户档案链接 | 创建客户时自动生成 | 客户基础信息 + 状态 + 关联行程报价单列表 | 手动复制粘贴微信 |
| **行程报价协作链接** ★（v0.9 合并） | **行程报价单"保存并通知"时自动生成** | **该版本行程+报价详情 + 修改备注 + 旅行社反馈区（行程+报价）+ 一手地接社编辑入口（按身份）** | **"保存并通知"自动复制到剪贴板，用户粘贴到微信** |

| **游客报价链接** | 旅行社加价后生成 | **对游客报价详情（选定车型）+ 行程**，不含旅行社报价 | 手动复制粘贴微信 |
| 沟通备注链接 | 手动添加备注时可生成 | 备注内容 + 上下文 | 手动复制粘贴微信 |

> **行程报价协作链接 vs 行程版本链接** ★（v0.9 更新）：
> - 行程报价协作链接：通过"保存并通知"生成，**链接+摘要自动复制到剪贴板**，H5 页面包含双向协作功能（旅行社可反馈行程+报价、一手地接社可编辑行程+报价），**一条链接含行程+报价** ★
> - 行程版本链接：系统自动生成的版本记录链接，H5 页面以展示为主，**不自动复制到剪贴板**
> - 两者指向同一行程报价单版本，但入口和交互能力不同

> **成本询价H5链接** ★（v0.9 新增）：
> - 通过"发送成本询价"生成，链接+摘要自动复制到剪贴板
> - H5 页面为**省地接社视角**：只显示行程需求（可编辑）+成本价填写区+行程编辑区+行程规划反馈区
> - 省地接社填写成本价后点击"保存并通知"→成本价提交+通知链接复制到剪贴板→粘贴到微信通知一手地接社
> - 成本询价H5链接不包含任何报价信息（对旅行社报价、利润、游客报价等），省地接社只能看到行程需求和填写成本价

> **三类协作链接用途严格区分（v0.9.12 澄清）**：
> - **行程报价协作链接（对客只读）**：由「保存并通知」在路线**未关联省地接社**时生成，发境外旅行社；只读，含行程+报价协作与反馈
> - **省地接社可编辑协作链接**：由「保存并通知」在路线**已关联省地接社**时自动改生成（或经「发起省地接社协作」），发省地接社；免登录、可编辑行程+填成本价
> - **成本询价 H5 链接**：由「发送成本询价」生成，发省地接社；聚焦成本填写
> - ⚠️ **防误发**：对客只读链接不可发给省地接社——省地接社会看到不可编辑、空白的页面（此即 v0.9.12 修复的现场问题）。v0.9.12 起系统按协作上下文自动选链，从源头避免误发

#### 4.4.4 链接访问与登录机制 ★（v0.9.1 简化：邀请制）

**准入流程（安全闭环 + 邀请制）** ★（v0.9.1 简化）：

| 步骤 | 访问者状态 | 系统行为 |
|---|---|---|
| 1. 管理者邀请 | 获得邀请链接 | 管理者发送邀请链接给员工（微信/短信） |
| 2. 微信授权登录 | 获取微信身份 | 首次打开H5完成微信授权登录 |
| 3. 判断归属 | 系统根据邀请链接判断归属 | 自动绑定到对应组织（旅行社/省地接社） |
| 4. 直接获得权限 | 正常使用 | 按邀请时分配的角色获得权限，登录态保持 30 天，邀请一次永久有效 |

> ★（v0.9.1 简化）**取消"待审核"状态**，改为"管理者邀请→员工微信授权→直接获得权限"。不再需要审核流程，首次打开H5完成微信授权登录即获得权限。

> **微信内无感登录**：
> - 旅行社/省地接社在微信中点开 H5 链接 → 微信 OAuth 自动识别身份 → 30天登录态有效 → **无需重复登录** ★（v0.9 扩展：省地接社也适用）
> - 首次打开需微信授权（一键授权，无需输入账号密码）
> - ★（v0.9.1 简化）通过邀请制获得权限后，后续在微信内打开任何 H5 链接均自动识别身份，**零门槛使用**

> **邀请规则** ★（v0.9.1 更新）：
> - 新旅行社/新省地接社首次接入：其管理者账号由一手地接社管理者邀请（一次性）
> - 旅行社员工：由本旅行社管理者邀请
> - 省地接社员工：由本省地接社管理者邀请 ★（v0.9 新增）
> - 邀请一次即可：通过微信授权后永久有效，30 天登录态到期重新授权不需要重新邀请
> - ★（v0.9.1 简化）取消审核流程，管理者邀请→员工微信授权→直接获得权限

> **安全铁律**：
> - 未获邀请的微信账号无法查看任何业务数据
> - 链接被转发给外人 → 外人微信授权后无邀请记录 → 无法访问
> - 一手地接社管理者可管理所有账号
> - 每个微信账号只能绑定一个旅行社或省地接社
> - ★（v0.9 新增）成本询价H5链接只有被分配的省地接社可访问，其他省地接社无法打开

> **设计原则**：
> - **零切换**：三方全程在微信内完成，不需要打开PC端、不需要安装App
> - **零门槛**：微信授权登录一次，30天免登，微信内打开H5自动识别身份
> - **有留痕**：每次"保存并通知"、每次反馈提交、每次成本价提交、每次链接打开都自动记录到沟通记录中心
> - **可追溯**：行程报价单版本 v1→v2→v3 每次修改都有变更备注，反馈意见有版本关联，成本报价有独立版本记录
> - **数据隔离**：旅行社看不到省地接社的成本价，省地接社看不到对旅行社报价，不同省地接社之间互相隔离 ★

### 4.5 沟通时间线（合并进路线详情 Tab3）(M4) ★（v0.9.9 由独立菜单合并）

> **v0.9.9 变更**：原 4.5「沟通记录中心」为独立一级菜单（与路线管理/知识库/账号管理并列），承载客户级沟通时间线。v0.9.9 取消该独立入口，将沟通时间线下挂到每条路线详情页，作为 **Tab3「💬 沟通时间线」**（与 Tab1「行程报价」、Tab2「客户信息」并列）。

>
> **保留能力**：浏览/类型筛选/手动备注/截图上传/全文检索/沉淀为知识/互动统计（H5打开次数、反馈次数、版本修改、保存并通知）完整保留在 Tab3 内，不降级。

### 4.2.17 核心业务规则决策汇总（v0.9.7 接受推荐方案）

> 本节汇总「核心业务规则决策清单 v0.2」中 **Q1.1–Q6.4 全部 24 项**的已决结论，作为开发阶段唯一参考。所有项均按推荐方案接受。

**（一）保存机制**
- 编辑操作采用 **自动保存草稿（乐观更新）**：编辑即落库 draft 态，无独立保存按钮；仅「保存并通知」转正并推送 H5。
- 自动保存后轻量提示：「已自动保存 · 刚刚」toast。
- 草稿与正式版为 **同一份数据 + status 标记**（draft / published），协作方只读 published 态。
- 保存失败：本地乐观更新 + 失败重试 + 顶部红条「保存失败，点击重试」。

**（二）行程 ↔ 报价联动**
- 加一天：**自动调整按量项数量**（包车天数+1、末尾加住宿则酒店晚数+1）+ 标红「需更新」，单价仍需人工复核。
- 删一天：按「行程结构」驱动反向调量（删中间天→包车-1；删住宿夜→酒店晚数-1，否则不变）。
- 改车型：**实时重算当前车型 5 级价**，其余车型成本保留。
- 「需更新」标记：**人工确认单价后手动清除**（点「复核完成」），不自动清除。
- 自定义报价项不参与天数联动，仅参与 5 级价计算。

**（三）版本号规则**
- 版本号 **仅「保存并通知」时 +1**（v3→v4）；草稿编辑不递增。
- 草稿独立标识：草稿「v3 · 草稿（未通知）」，正式版「v3」。
- 协作方（旅行社/省地接社）**仅见最后正式版**；草稿仅一手可见。
- **MVP 不支持版本回滚**，仅保留版本历史列表。

**（四）权限矩阵（三方角色）**
- 一手地接社：全权限（增删改查全部）。
- 旅行社：看行程 + 提反馈 + 改「自己负责的客户字段」；**不可改成本价①②**；可见 ③④⑤ 及客户联系信息；**不可见成本价①②**。
- 省地接社：仅成本询价 H5，只能填 ①成本价，**不可见 ②③④⑤ 及客户信息**。
- 删除路线/客户档案：**仅一手**有权。

**（五）校验与异常分支**
- 删除最后一天：硬拦截（至少保留 1 天）。
- 清空报价（0 项）：**不允许**；「保存并通知」前校验至少 1 项，否则阻断。
- 并发编辑冲突：反馈（评论）与编辑（数据）不冲突；同字段同时编辑「后写覆盖 + 版本提示」；MVP 不锁。
- 必填项（行程标题、报价成本）：保存草稿不拦（允许半成品）；「保存并通知」前强制校验。
- 数字合法性：输入层即时拦截非数字/负数；0 人天允许（待定团）。

**（六）扩展 Gap**
- 多语言翻译：保存并通知时自动翻译（TMT）生成 EN/ZH/TH/RU 副本，手动可重翻。
- 成本询价闭环：省地接社提交 → 一手收通知 → 一键「采纳」回填①成本价 → 触发需更新。
- 通知机制：复制链接 + 唤起微信分享（或企业微信消息）；原型展示「复制链接」。

**（七）链接生成与幂等（v0.9.12 新增）**
- 「保存并通知」按协作上下文生成链接：路线未关联省地接社 → 对客只读「行程报价协作链接」；路线已关联省地接社 → 免登录可编辑「省地接社协作链接」。二者不可混用，对客链接误发省地接社会导致其看到不可编辑空白页。
- 省地接社协作链接幂等：同一路线 + 同一省地接社始终复用唯一 token 与唯一成本询价记录；重复点「保存并通知」/「发起省地接社协作」不生成新链接、不新增询价记录。
- 客户档案依附路线：客户档案与路线 1:1 绑定，**无孤儿客户**；删除路线级联删客户。

### 4.2.18 利润表达模式（含利润直接报价）

> **关联决策**：Q7.1（推荐方案，待产品负责人最终确认后生效）
> **问题来源**：一手地接社自行行程和报价时，是否必须强制按「成本价 + 利润」分解录入？能否选择「已含利润」——即一手地接社直接报一个包含成本价和利润的售价？

**（一）核心判断**

1. **数据层（强制分离，不可省略）**：5 级报价引擎必须始终可计算并存储「成本价 ①」与「利润 ②」两个值，无论用户如何输入。理由：
   - 利润核算/报表：一手地接社必须知道每条路线、每个周期的毛利率；
   - 下游链路一致性：③ = ①+② 必须成立，④ 旅行社加价、⑤ 对游客报价都建立在 ③ 之上；
   - 省地接社询价闭环：① 含通过 H5 向省地接社询得的成本，必须可追溯；
   - 审计/版本一致：每一级价格应由前级推导，避免数字对不上。
   - → **「成本与利润在系统里必须分离计算」是硬性要求，不能省。**

2. **输入层（可选，用户选择）**：用户**不必**把成本和利润拆成两个数字分别录入。提供两种输入模式并存：

   | 模式 | 录入方式 | 系统推导 | 适用场景 |
   |---|---|---|---|
   | **模式 A 成本+利润分解**（现有默认） | ① 成本价 + ② 利润（金额或 %） | ③ = ①+② | 与旅行社谈毛利率；省地接社成本需可见；控制批发价 |
   | **模式 B 含利润直接报价**（新增） | ① 成本价 + 直接录入 ③ 对一手报价（「已含利润」打包售价） | ② 利润 = ③ − ①（展示并可编辑） | 一手地接社习惯「这段我卖给社 5000 全包」；FIT 直客/打包产品；供应商给净含利润价 |

   > **关键点**：模式 B 下利润**仍被计算并存储**，只是由售价反推而非直接录入。所以「必须分离成本+利润」的数据要求被满足，改变的只是输入交互。

3. **延伸 — 直客场景（无旅行社）**：当一手地接社直接卖游客（无旅行社中间层），③→④→⑤ 的社链条塌缩，此时一手地接社的「含利润报价」即 ⑤。引擎仍追踪 ① 与反算 ② 供报表。

**（二）护栏**
- 模式 B 反算利润若 **< 0**（售价低于成本），系统标红「亏损报价」并允许人工覆盖（如引流亏损单），但报表中单独标记。
- 模式按 **路线级** 选择 A / B；MVP 先不支持按报价项混合（后续可扩展）。

**（三）原型/P​RD 同步动作（确认后执行）**
- 报价区块 header 增加「利润模式：A 分解 / B 含利润」切换；
- 模式 B 时，② 利润字段变为只读（由③反算），③ 变为可输入；
- PRD 5 级报价引擎章节补充两种模式的字段定义与校验。

### 4.8 案例展示（获客 · 宣传闭环）(MVP 一期交付) ★（v0.9.8 新增规划 → v0.9.11 拉回 MVP）

> **v0.9.8 新增模块规划**：当前产品为纯 B2B 协作后台工具。案例展示把已交付/已确认的定制行程脱敏后沉淀为可分享作品集，对外通过公开 H5（免登录）在微信生态传播获客，对内沉淀品牌资产。与路线管理共用同一「路线」数据脊柱——案例是路线的"发布态派生"，复用现有 H5 链接 / 二维码 / 海报能力，开发代价小。**v0.9.11 起纳入 MVP 一期交付**（架构设计 v1 澄清闸门确认），与路线管理 / 知识库 / 账号管理同期上线；发布前须通过脱敏合规校验（见 4.8.6）。

#### 4.8.1 定位与价值

- **获客**：潜在游客 / 下游分销渠道浏览精美历史案例，种草后发起"类似行程咨询" → 生成新线索路线进入路线管理协作流
- **宣传**：每条成功交付的行程都可"发布为案例"反哺案例库，形成社会证明，越用越多、越强
- **资产沉淀**：把分散在微信里的"成功案例"结构化留存，不再随人员流动流失

#### 4.8.2 信息架构

- **新菜单项 🌟 案例展示**（与路线管理 / 知识库 / 账号管理并列，菜单 4 项）
- **一手地接社视角（管理）**：案例管理列表（状态：草稿 / 已发布 / 已下线）+「发布新案例」入口
- **对外展示（公开 H5，免登录）**：每个案例一个独立公开页，含封面图、目的地、天数、主题标签、亮点卖点、脱敏行程概览、参考价格区间（或"定制询价"）、分享数据（浏览 / 分享 / 咨询）
- **发布流**：路线管理详情 →「发布为案例」→ 选封面 / 写卖点 / 系统自动脱敏 → 生成可分享 H5 + 二维码 + 海报

#### 4.8.3 四段增长闭环

| 阶段 | 动作 | 系统支撑 |
|---|---|---|
| ① 获客 | 案例 H5 在微信 / 朋友圈 / 公众号传播，浏览种草 | 公开 H5 + 二维码 + 海报 |
| ② 定制 | 案例页「咨询类似行程」→ 生成新线索路线（状态：新线索） | 路线管理新建路线 + 协作流 |
| ③ 服务 | 行程报价三方协作 → 出行交付 | 路线管理 + H5 协作闭环 |
| ④ 宣传 | 完成行程「发布为案例」→ 反哺案例库 | 案例展示发布流 |

> 同一「路线」实体贯穿四段——案例只是路线的发布态派生，闭环自增强：交付越多 → 案例越丰富 → 获客越强。

#### 4.8.4 脱敏与合规约束（Non-goals，见 2.2）

- 案例**必须脱敏**：客户真名、证件号、精确出行日期、客户专属合同价不得公开；仅目的地 / 天数 / 主题 / 亮点 / 参考价格区间可展示
- 遵守《个人信息保护法》；发布前由发布人确认脱敏，系统提供脱敏默认值与校验
- v1 不做：在线支付 / 游客自助下单 / 图文混排富文本 CMS（仅结构化案例卡）

#### 4.8.5 指标（见 6.3 案例展示指标）

案例展示 UV/PV、分享次数、案例→咨询转化率、案例库规模 / 月新增、获客成本节约（远期）

#### 4.8.6 合规发布校验（脱敏发布流）★（v0.9.11 新增，MVP 一期强约束）

> 案例展示纳入 MVP 一期后，发布即对外可见，脱敏合规为硬门槛（对应架构设计 v1 风险项 ①）。

- **仅允许从「已确认 / 已成单」路线派生案例**：草稿、待报价、咨询中路线不可发布，确保展示内容均为真实交付
- **发布前强制脱敏校验开关**：系统自动检测并拦截以下字段外泄——客户真名、证件号、精确出行日期（仅保留月份/季节）、客户专属合同价（仅保留参考价格区间）
- **发布人二次确认**：系统提供脱敏默认值，发布人须手动勾选"已确认脱敏合规"方可发布；保留发布审计日志
- **可下线 / 可编辑**：已发布案例支持下线（对外 404）与重新编辑后重发；下线不影响历史线索归属
- **合规依据**：遵守《个人信息保护法》；禁止展示任何可定位到具体个人的信息（见 2.2 Non-goals 对应条目）

### 5.4 旅行社加价流程

```
[旅行社收到一手地接社报价]
  → 旅行社销售在 H5 页面查看行程+报价
  → 查看分车型方案对比（7座/9座/14座各自报价）
  → 选择确认车型方案（或暂不选择，全部报给游客）
  → 点击"加价生成游客报价"
  → 选择加价方式（默认"按固定金额" ★v0.9.1）：
      ├── 按固定金额（默认）：输入金额（如¥3,000）→ 加价 = ¥3,000
      └── 按百分比加成（需手动切换）：输入百分比（如10%）→ 加价 = 旅行社报价 × 10%
  → 系统自动计算对游客报价 = 旅行社报价 + 加价
  → 生成游客报价快照（冻结，不可修改）
  → 可选操作：
      ├── 生成游客版 H5 链接 → 转发给游客
      ├── 下载游客版 PDF（不含旅行社报价/成本价/利润）
      └── 一键转发摘要到微信

  → 系统自动通知一手地接社："旅行社已加价生成游客报价"
  → 一手地接社可查看：加价方式、加价金额、游客报价
  → 路线管理（看板视图）更新该路线待办为"已确认" ★
  → 沟通记录中心自动记录加价行为

  若游客要求改价：
  → 旅行社重新加价 → 生成新的游客报价快照
  → 原快照保留（历史追溯）
```
