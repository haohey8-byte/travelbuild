# 上线操作清单（傻瓜版）—— PandaKing9 入境游定制协作工作台

> 配套文档：`doc/06-上线部署.md`（技术细节）、`doc/05-数据库与部署.md`（迁移/种子）。
> 本文是「你点哪里、填什么、把什么发我」的逐步清单。
>
> 图例：
> - 【你】必须你本人登录腾讯云控制台操作（涉及账号登录态 / 实名 / 付费确认，AI 无法代点）
> - 【AI】由我在 WorkBuddy 沙箱里代跑（代码在这里）
> - 【你+AI】需要你提供信息（连接串 / 密钥 / 域名），我再代跑命令

---

## 第 0 步 · 把代码推到 Git 远程（【你】建仓库，【AI】代推）

CloudBase 云托管是「绑定 Git 仓库 → 自动构建镜像」，所以代码必须在一个 CloudBase 能访问的 Git 仓库（GitHub / 腾讯工蜂 / CODING / GitLab）。

1. 【你】在 GitHub 或腾讯工蜂新建一个**空仓库**（不要勾初始化 README / .gitignore）。
2. 【你+AI】把仓库地址（HTTPS 或 SSH）发给我 → 我代跑：
   ```bash
   git remote add origin <你的仓库地址>
   git push -u origin main
   ```
   （当前沙箱代码已提交到本地 git，push 即可，无需你动代码。）

---

## 第 1 步 · 开通腾讯云账号 + CloudBase（【你】）

1. 打开 https://cloud.tencent.com 注册账号并完成**实名认证**（个人/企业均可）。
2. 进入「云开发 CloudBase」控制台，按指引开通服务；确保「云托管」能力已开启。

---

## 第 2 步 · 建 PostgreSQL 实例（【你】建，【你+AI】拿串）

1. 【你】CloudBase 控制台 → 数据库 → 新建 **PostgreSQL** 实例（记一下地域，如 `ap-shanghai`）。
2. 【你】实例就绪后，复制其连接串，形如：
   ```
   postgresql://<user>:<password>@<host>:5432/pandaking9?schema=public&sslmode=require
   ```
   ⚠️ CloudBase Postgres 默认要求 SSL，务必保留 `sslmode=require`。
3. 【你+AI】把**整串连接串**发我 → 我写入 `backend/.env` 的 `DATABASE_URL` → 代跑：
   ```bash
   pnpm -F backend prisma:deploy   # 建表（应用 migrations/0_init）
   pnpm -F backend db:seed         # 写入演示数据：一手/旅行社/省地接社/演示邀请
   ```

---

## 第 3 步 · CloudBase 云托管（【你】点，配置我给）

1. 【你】云托管 → 新建服务（例如 `pandaking9-backend`）。
2. 【你】代码源 → 绑定**第 0 步**的 Git 仓库与分支（如 `main`）。
3. 【你】构建配置（照抄，已验证）：
   | 项 | 值 |
   |---|---|
   | Dockerfile 路径 | `backend/Dockerfile` |
   | 构建上下文目录 | `/`（仓库根，因 pnpm workspace 在根） |
   | 监听端口 | `3000` |
   | 健康检查路径 | `/api/health` |
4. 【你】环境变量（逐个新建）：
   | 变量 | 值 |
   |---|---|
   | `DATABASE_URL` | 第 2 步的连接串 |
   | `JWT_SECRET` | 随机长串，例如本地跑 `openssl rand -hex 32` 生成 |
   | `DEV_BYPASS_AUTH` | `false`（生产必须关掉开发旁路） |
   | `WECHAT_APPID` | 留空（先走演示账号，后续再填） |
   | `WECHAT_SECRET` | 留空 |
   | `PORT` | `3000`（云托管通常自动注入，可不填） |
5. 【你】点「部署」，等镜像构建完成（几分钟，首次拉依赖较慢）。
6. 【你+AI】把云托管分配的**后端域名**发我（或贴构建日志），我帮你跑端到端验证。

---

## 第 4 步 · 前端构建 + 上传 COS + EdgeOne（【AI】上传，【你】配域名）

1. 【AI】我代跑前端生产构建（已验证可编译）→ 产物 `frontend/dist`：
   ```bash
   pnpm -F frontend build
   ```
2. 【你+AI】你把 **COS 密钥**发我（4 项）→ 我代跑上传脚本：
   ```bash
   # 需先装一次依赖：pnpm add -w cos-nodejs-sdk-v5
   COS_SECRET_ID=xxx COS_SECRET_KEY=xxx COS_BUCKET=xxx COS_REGION=ap-shanghai \
     node deploy/deploy-frontend.mjs
   ```
   脚本会把 `frontend/dist` 全量上传到 COS（`index.html` 不缓存、带 hash 资源长缓存）。
3. 【你】EdgeOne → 接入加速域名（如 `app.xxx.com`），源站选上面的 COS Bucket。
4. 【你】配两条规则（关键）：
   - **SPA 回源重写**：所有非文件请求（无扩展名）→ 回源 `/index.html`，否则刷新 `/routes/kanban` 会 404。
   - **API 反代**：`/api/*` → 回源到 CloudBase 云托管后端域名。这样前端无需改配置且规避跨域。
   - （若不做反代，也可在 `frontend/.env.production` 设 `VITE_API_BASE=<云托管域名>` 后重新 build；但推荐反代。）

---

## 第 5 步 · 端到端验证（【你+AI】）

逐项确认（任一项异常把日志/截图发我）：
- [ ] `GET <云托管域名>/api/health` → `200 {"status":"ok"}`
- [ ] 浏览器打开 `<CDN域名>/` → 前端首页（SPA 壳）
- [ ] 刷新 `/routes/kanban` 不 404（SPA 重写生效）
- [ ] 新建路线 → 「保存并通知」→ 复制 H5 链接 → 匿名打开 `/h5/route/:token` 看到脱敏行程与对客价
- [ ] H5 提交反馈 → 后端 `RouteFeedback` 落库
- [ ] CloudBase Postgres 里 `Route` / `CostInquiry` 等表已存在（迁移成功）

## 你需要发给我的信息汇总

| 信息 | 用途 | 我拿到后做什么 |
|---|---|---|
| Git 远程仓库地址 | 云托管绑定构建 | 代跑 `git push` |
| PostgreSQL 连接串 | 建表 + 种子 | 代跑 `prisma:deploy` + `db:seed` |
| COS 密钥（4 项：ID/Key/Bucket/Region） | 前端上传 CDN | 代跑 `deploy-frontend.mjs` |
| 云托管 / EdgeOne 域名（可选） | 端到端核对 | 代跑健康检查与联调 |

> 安全提示：以上密钥仅用于本次部署，建议在腾讯云「访问管理」创建**仅含必需权限**的子账号密钥，部署完成后可禁用。不要将密钥提交进代码仓库（`.gitignore` 已忽略 `.env`）。
