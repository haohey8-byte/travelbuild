## 第 1 步：本机推送代码（3 笔提交）

### 方法 A：Git Credential Manager（本机有浏览器，推荐）

```powershell
# 1) 进入项目目录（换成你本机实际路径）
cd D:\workbuddy-project

# 2) 先核对：应该列出 3 笔提交
git log --oneline origin/master..HEAD

# 3) 推送。若弹出 GitHub 登录框，用浏览器授权即可
git push origin master

# 4) 验证：下面应该为空，说明全部上去了
git log --oneline origin/master..HEAD
```

### 方法 B（备选）：本机也弹不出登录框（无头 / 服务器环境）

用 Personal Access Token 替代 OAuth：

```powershell
# 1) GitHub → Settings → Developer settings → Personal access tokens 生成一个有 repo 权限的 token
# 2) 用 token 重写 remote（把 <用户名>/<仓库> 换成你的实际地址）
git remote set-url origin https://<你的用户名>:<TOKEN>@github.com/<owner>/<repo>.git
git push origin master
# 3) 推送成功后，把 remote 改回普通 https（避免 token 留在 config 里）
git remote set-url origin https://github.com/<owner>/<repo>.git
```

> 也可改用 SSH：`git remote set-url origin git@github.com:<owner>/<repo>.git`，前提是你本机已配好 SSH key。

## 第 2 步：确认生产库迁移已应用（关键，幂等）

后端容器启动不自动跑迁移，必须库结构先就绪，否则 API 会 500。**若沙箱之前已成功跑过可跳过；不确定就重跑一次**（已应用的迁移不会重复执行）。

> ⚠️ 用**本地二进制**跑，别用 `pnpm prisma …`——pnpm 会改写 `pnpm-workspace.yaml` 并留 `_tmp_*` 垃圾文件（已踩过坑）。

```powershell
cd D:\workbuddy-project\backend

# 确保本机 .env 里有生产 DATABASE_URL，然后：
.\node_modules\.bin\prisma migrate deploy
.\node_modules\.bin\tsx prisma/seed.ts      # 仅首次建库需要；已 seed 过可省略
```

成功标志：`migrate deploy` 输出 `2 migrations found / all migrations have been applied` 之类。

## 第 3 步：重新部署后端（CloudBase 云托管）

这是**网页控制台**操作，不是命令行：

1. 登录 CloudBase 控制台 → 找到你的云托管服务（后端）。
2. 确认该服务「代码来源」绑定的是这个 GitHub 仓库的 `master` 分支。
3. 点 **重新部署 / 重新构建**（确保它拉的是第 1 步刚推上去的最新 master）。
4. Dockerfile 路径填 **`backend/Dockerfile`**（构建阶段会自动 `pnpm install` + `prisma generate` + `nest build`）。
5. **检查环境变量**：确认 `DATABASE_URL`、`DEV_BYPASS_AUTH`（演示期）、`JWT_SECRET` 等已在 CloudBase 环境变量里配好，不要动它们。
6. 部署完成后，云托管会给你一个新的访问域名（或复用旧的）。

> 容器启动后访问 `https://<你的后端域名>/api/health`，应返回 200。

## 第 4 步：重新部署前端（COS + EdgeOne）

### 4.1 本机构建前端

```powershell
cd D:\workbuddy-project
```

- **标准方式**：
  ```powershell
  pnpm -F frontend build
  ```
- **稳妥备选（绕过 workspace，纯 vite，避免 pnpm 污染 `pnpm-workspace.yaml`）**：
  ```powershell
  cd frontend
  .\node_modules\.bin\vite build
  cd ..
  ```

成功标志：生成 `frontend/dist/index.html` 及带 hash 的静态资源。

> ⚠️ 踩坑：若 `pnpm` 中途报错并发现 `pnpm-workspace.yaml` 被改、根目录出现 `_tmp_*` 空文件，执行：
> ```powershell
> git restore pnpm-workspace.yaml
> Remove-Item _tmp_* -ErrorAction SilentlyContinue
> ```
> 然后改用上面的「稳妥备选」纯 vite 命令。

### 4.2 上传到 COS

首次部署需先装 COS SDK（若已装可跳过）：
```powershell
pnpm add -w cos-nodejs-sdk-v5
```

设置 4 个环境变量（建议放本地 shell 或 CI 密钥，**不要入库**），然后上传：
```powershell
$env:COS_SECRET_ID="你的SecretId"
$env:COS_SECRET_KEY="你的SecretKey"
$env:COS_BUCKET="你的Bucket"
$env:COS_REGION="你的Region"   # 如 ap-guangzhou

node deploy/deploy-frontend.mjs
```
成功标志：输出 `上传完成。请在 EdgeOne 控制台配置加速域名与 SPA 回源重写。`

### 4.3 EdgeOne（控制台，基本不用动）

- SPA 回源重写：`/* → /index.html`（保证前端路由刷新不 404）。
- `/api` 反代到 CloudBase 后端域名。
- 这两项首次上线时配过，后续发布**不用每次改**。

## 常见问题速查

| 现象 | 原因 | 处理 |
|---|---|---|
| `git push` 卡住无输出 | 沙箱/无头环境 GCM 想弹浏览器 OAuth 弹不出 | 用本机（有浏览器）推；或改用 PAT / SSH（见第 1 步方法 B） |
| 部署后 API 返回 500 / 表不存在 | 生产库没跑迁移 | 第 2 步重跑 `migrate deploy` |
| `pnpm` 后 `pnpm-workspace.yaml` 被改、根目录出现 `_tmp_*` | pnpm 副作用 | `git restore pnpm-workspace.yaml` + 删 `_tmp_*`；改用本地二进制 / 纯 vite 命令 |
| `deploy-frontend.mjs` 报缺 `cos-nodejs-sdk-v5` | 依赖未装 | `pnpm add -w cos-nodejs-sdk-v5` |
| 上传 COS 报缺环境变量 | 4 个 COS_* 未设置 | 按 4.2 设好再跑 |
| PDF 下载出来是空白 | 离屏渲染容器未挂载 | 检查 `RoutePdf` 组件是否在页面内（v-if 控制，导出时已挂载） |
