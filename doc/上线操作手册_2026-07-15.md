> 2. 常态化：后端 `Dockerfile` 容器启动自动 `prisma migrate deploy`（幂等），**以后每次重部署后端都会自动应用已提交的新迁移**。

> 前端部署密钥放 `deploy/.env.local`（gitignored），用一键脚本 `deploy/deploy.sh` 固化。

## 一、以后每次部署（固定 2 步）

> 前提：代码已改完、本地编译通过（后端 `nest build`、前端 `vue-tsc --noEmit && vite build`）。

### 第 1 步：控制台重新部署后端（网页，约 3 分钟）

1. 登录 [CloudBase 控制台](https://console.cloud.tencent.com/tcb) → **云托管** → **服务列表**。
2. 找到后端服务 → 点 **重新部署 / 重新构建**（拉最新 master）。
3. 确认：**Dockerfile 路径 = `backend/Dockerfile`**，**构建上下文 = 仓库根**。
4. 环境变量保持不动：`DATABASE_URL` / `DEV_BYPASS_AUTH=true` / `JWT_SECRET`。
5. 部署完成 → 容器启动自动 `prisma migrate deploy`（已应用的迁移不会重跑）→ 启动 NestJS。
6. 验证：`curl -s https://<后端域名>/api/health` → 期望 `200`。

### 第 2 步：一键部署前端（本机终端，1 条命令）

```bash
cd /d/workbuddy-project
bash deploy/deploy.sh
```

脚本自动完成：① `git push origin master` ② `npm install && npm run build` ③ `node deploy/deploy-frontend.mjs`（读 `deploy/.env.local` 里的密钥）。
成功标志：输出「部署完成」。

> 首次使用需先准备 `deploy/.env.local`（见第三节）。

## 三、部署密钥（SecretId / SecretKey）准备

### 3.1 去哪拿（详细步骤）

`COS_SECRET_ID` / `COS_SECRET_KEY` 本质就是**腾讯云 API 密钥**，不是 CloudBase 专属，是整个腾讯云账号的密钥。`deploy-frontend.mjs` 用它调 `tcb` API 把前端产物上传到静态托管。

获取步骤：

1. 用**部署这个 CloudBase 环境（`travelbuild-d3gvgvtj70ddd0e43`）的那个腾讯云账号**登录。
2. 打开 [访问管理 → API 密钥](https://console.cloud.tencent.com/cam/capi)（即控制台右上角头像 → **访问管理** → 左侧 **访问密钥 → API 密钥管理**）。
3. 页面列出密钥：
   - **SecretId**：明文，形如 `AKIDxxxxxxxxxxxxxxxxxxxxxxxx`，直接复制。
   - **SecretKey**：默认隐藏，点 **显示** 才可见（32 位字符串）。
4. 若一个密钥都没有：点 **新建密钥**，立即生成一对，复制保存。
5. ⚠️ **SecretKey 只在创建 / 显示时可见，等同账号密码，勿泄露、勿提交 git**。若是子账号（协作者），需主账号授予 CloudBase / COS 相关权限。

### 3.2 写入本地文件（gitignored，不入库）

```bash
cd /d/workbuddy-project
cp deploy/env-local.example deploy/.env.local
# 用编辑器打开 deploy/.env.local，把 SecretId / SecretKey 填进去保存
```

`deploy/.env.local` 内容（已被根 `.gitignore` 的 `.env.*` 规则忽略，绝不会被提交）：

```bash
export COS_SECRET_ID="AKIDxxxxxxxxxxxxxxxxxxxxxxxx"
export COS_SECRET_KEY="xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
export TCB_ENV_ID="travelbuild-d3gvgvtj70ddd0e43"
export HTTPS_PROXY="http://127.0.0.1:7897/"
export HTTP_PROXY="http://127.0.0.1:7897/"
```

> 换机器 / 换账号时，只改这个文件即可，部署命令不用动。

## 四、以后新增数据库迁移怎么办

本机连不上生产库（云托管内网免密账号本机不可认证），`prisma migrate dev` 本机也跑不起来。新增迁移用「手写迁移 + 容器自动应用」：

1. 在 `backend/prisma/migrations/<时间戳>_<名称>/migration.sql` 手写增量 SQL（参考现有 `ALTER TABLE` 写法）。
2. `git push` → 控制台重新部署后端 → 容器自动 `prisma migrate deploy` 应用这条新迁移。
3. 若必须手动在 SQL 编辑器执行（如数据迁移），执行后**务必补 `_prisma_migrations` 记录**（用 `gen_random_uuid()` + 该迁移文件的 sha256 checksum），否则以后容器会重复执行而报错。

## 六、常见问题速查

| 现象 | 原因 | 处理 |
|---|---|---|
| `git push` 报 `Empty reply from server` | GitHub 直连被重置 | 确保 `deploy/.env.local` 里有 `HTTPS_PROXY`，或手动 `export HTTPS_PROXY="http://127.0.0.1:7897/"` |
| 后端部署后容器反复重启 | 迁移失败（如重复 ALTER 列已存在） | 查 `_prisma_migrations` 是否已记录该迁移；手动执行 SQL 后补记录 |
| `deploy-frontend.mjs` 报缺 `COS_SECRET_ID` | 没加载 `.env.local` | 在 `deploy/` 目录运行 `bash deploy/deploy.sh`，或手动 `source deploy/.env.local` |
| `npx tcb` 下载失败 | npm 网络 / 代理 | 确认 `HTTPS_PROXY` 已设；或先 `npm install -g @cloudbase/cli` |
| 看板空白、路线显示不全 | 浏览器缓存了旧 token | 清 `localStorage` 旧 token + 硬刷新 |
