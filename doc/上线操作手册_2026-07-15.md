## 第 1 步：本机推送代码（如果还有未推送的提交）

### 方法 A：本机有代理客户端（推荐）

```powershell
cd D:\workbuddy-project
$env:HTTPS_PROXY="http://127.0.0.1:7897/"
git push origin master

# 验证：下面应无输出，说明全部上去了
git log --oneline origin/master..HEAD
```

### 方法 B：无代理/无头环境

用 GitHub Personal Access Token 替代 OAuth：

```powershell
# 1) GitHub → Settings → Developer settings → Personal access tokens → 生成一个有 repo 权限的 token
# 2) 用 token 重写 remote（把 <owner>/<repo> 换成你的实际地址）
git remote set-url origin https://<你的用户名>:<TOKEN>@github.com/<owner>/<repo>.git
git push origin master
# 3) 推送成功后，把 remote 改回普通 https（避免 token 留在 config 里）
git remote set-url origin https://github.com/<owner>/<repo>.git
```

> 如果之前 git push 报 `Empty reply from server`，这是 GitHub 直连被 TLS 重置，用方法 A 的代理即可。

## 第 2 步：生产库跑迁移（migrate deploy + seed）

后端容器启动**不自动**跑迁移，必须库结构先就绪，否则 API 会 500。`migrate deploy` 是幂等的，已应用的迁移不会重复执行。

### 2.1 获取 `DATABASE_URL`

- CloudBase 控制台 → **数据库** → 你的 PostgreSQL 实例 → **连接串 / 外网地址**。
- 如果只有内网地址，本机连不上 → 先点 **开启外网地址 / 公网访问**，等几分钟后复制外网连接串。
- 格式：
  ```
  postgresql://<用户名>:<密码>@<主机>:<端口>/<数据库名>?schema=public
  ```
- 用户名、密码、数据库名在控制台「数据库设置 / 账号管理」里查看，不要写死在本手册中。

### 2.2 执行迁移

```powershell
cd D:\workbuddy-project\backend

# 把下面的 <用户名>、<密码>、<数据库名> 替换成你的实际值
$env:DATABASE_URL="postgresql://<用户名>:<密码>@sh-postgres-k1g3uqg8.sql.tencentcdb.com:25183/<数据库名>?schema=public"

# 方式 A：本地已安装 backend 依赖（有 node_modules）
.\node_modules\.bin\prisma migrate deploy
.\node_modules\.bin\tsx prisma/seed.ts

# 方式 B：零安装，用 npx 临时拉取（版本与项目锁定一致）
npx prisma@5 migrate deploy
npx tsx@4 prisma/seed.ts
```

成功标志：

```
2 migrations found in prisma/migrations
2 migrations have been applied
```

其中应包含 `20260716000000_agency_model`。

> 迁移 SQL 里已用 `INSERT ... ON CONFLICT` 插入了两个演示机构：`org-agency-seed`（环球旅行社）和 `org-provincial-seed`（川内地接社），所以即便不跑 `seed` 这两个机构也已存在。`seed` 还会创建演示用户/路线，幂等可跑可不跑。

## 第 3 步：CloudBase 控制台重新部署后端

这是网页操作：

1. 登录 [CloudBase 控制台](https://console.cloud.tencent.com/tcb) → **云托管** → 服务列表。
2. 找到后端服务，确认「代码来源」绑定的是 GitHub 仓库 `master` 分支。
3. 点 **重新部署 / 重新构建**（会拉取第 1 步刚推上去的最新 master）。
4. 确认 Dockerfile 路径 = **`backend/Dockerfile`**，构建上下文 = 仓库根。
5. 检查**环境变量**不要动：
   - `DATABASE_URL` = 生产库连接串
   - `DEV_BYPASS_AUTH=true`（演示期）
   - `JWT_SECRET`
6. 部署完成后，CloudBase 会给出新的访问域名（或复用旧域名）。

验证后端：

```powershell
Invoke-RestMethod -Uri "https://<你的后端域名>/api/health" -Method GET
# 期望返回 200
```

> 容器里只执行 `node backend/dist/main.js`，**不会自动 migrate deploy**，所以第 2 步必须先做。

## 第 4 步：前端构建与部署（CloudBase 静态网站托管）



成功标志：输出「部署完成」。脚本会自动执行 `npx tcb login --apiKeyId ... --apiKey ...` 然后 `npx tcb hosting deploy frontend/dist /`。

> 这些密钥在腾讯云控制台 → **访问管理 → 访问密钥（API 密钥）** 里新建。不要在代码仓库里提交真实密钥。

## 常见问题速查

| 现象 | 原因 | 处理 |
|---|---|---|
| `git push` 报 `Empty reply from server` | GitHub 直连被重置 | 设 `$env:HTTPS_PROXY="http://127.0.0.1:7897/"` 再推 |
| 后端部署后 API 返回 500 / 表不存在 | 生产库没跑迁移 | 第 2 步重新执行 `migrate deploy` |
| `.
ode_modules\.bin	sx` 不存在 | 没装 backend 依赖 | 改用 `npx prisma@5` / `npx tsx@4`，或 `npm install` 后再试 |
| `deploy-frontend.mjs` 报 `npx tcb` 下载失败 | npm 网络或代理问题 | 先设 `$env:HTTPS_PROXY="http://127.0.0.1:7897/"` 再执行；或先 `npm install -g @cloudbase/cli` |
| 看板空白、路线显示不全 | 浏览器缓存了旧 token | 清 localStorage 旧 token + 硬刷新 |
| 创建路线提示「无权限/必须选旅行社」 | 角色归属逻辑生效 | 一手选旅行社；旅行社自动归属；省地接社不能创建 |
