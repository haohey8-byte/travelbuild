# 省地接社 H5 页面：按钮合并与页面边界的产品设计决策

> 关联代码：`frontend/src/views/H5ProvincialRoute.vue`
> 关联页面模型（后端）：`GET /h5/route/:token`（协作数据）、`POST /h5/route/:token/edit`（省地接社保存）、`GET /share/route/:token`（SSR 对客 H5，由 `H5Route.vue` 消费）

## 问题 1：省地接社 H5 的「保存并通知（同步一手）」与「提交建议」是否应合并？

### 结论：**合并**。两个按钮合并为单一动作「保存并回传一手」。

### 判断理由（产品设计逻辑）

1. **协作本质是一次"回传"事件**。省地接社对一手的产出是单次的：编辑当地行程 + 回填成本① +（可选）说明。把它拆成"保存"和"提交建议"两个独立按钮，会让用户产生歧义——
   - 点「保存并通知」时，说明有没有发出去？
   - 点「提交建议」时，成本①有没有保存？
   这正是用户此前反复遇到的"按钮语义不清"类问题。
2. **两个动作最终都同步一手**，目标一致，没有必须分开的业务约束**（成本①是正式交付物，说明是其附件说明，不是独立工作流）**。
3. **与一手控制台语义一致**：一手在 kanban 用「保存并通知」一次性交付，供应商侧也用单一动作交付，符合系统既有心智模型。
4. **降低认知负荷与出错面**：少一个按钮、少一条分支路径，符合"最小交互、最少疑惑"的协作工具设计原则。

### 实现要点（已落地，commit `73024ae`）

- 删除独立的 `onSubmitFeedback` 与「提交建议」按钮区块。
- `onSave` 重命名为 `onSubmitHandoff`，合并逻辑：先 `editH5ProvincialRoute`（行程+成本①），若说明非空则一并 `submitH5Feedback`；说明提交失败时仅提示、不阻断成本保存。
- "回传说明（可选）"输入框上移到主按钮上方，明确是本次回传的附件。
- 已回传说明/建议历史列表保留并改名「已回传的说明 / 建议」。

## 问题 2：省地接社完成报价后生成的 H5 页面，是否应与一手（对客）的 H5 页面统一为一个页面？

### 结论：**不合并，保持两个独立页面**。本次不做修改。

### 判断理由（产品设计逻辑）

系统存在两类 H5，受众与数据可见性根本不同：

| 维度 | 对客 H5（`/share/route/:token` → `H5Route.vue`） | 省地接社协作 H5（`/h5/provincial-route/:token` → `H5ProvincialRoute.vue`） |
|---|---|---|
| 受众 | **终端客户**（付费方） | **省地接社**（供应商） |
| 交互 | 只读查看 + 简单反馈 + 导出 PDF | 可编辑行程 + 填成本① + 写说明 |
| 可见数据 | 仅"对客总价"，**绝不可见任何成本/利润** | 可编辑"成本①"，**绝不可见对客价/利润②** |

1. **数据隔离是核心约束，合并会直接破坏它**。项目既有的三角色物理隔绝原则（agency ↔ provincial 互不可见，客户只见对客价）是用户此前明确确立的。把两个页面合并，要么把成本①暴露给客户（灾难性），要么把对客价暴露给供应商（违背隔离），两者皆不可接受。
2. **交互模型不兼容**。对客页是"只读展示 + 轻反馈"；供应商页是"富编辑器（增删天数、改景点、填成本表）"。合并后会得到一个对客户过于复杂、对供应商又泄露成本的四不像页面。
3. **两者已正确共享设计语言**：视觉风格（`.h5`）、工具函数（`collabNotifyText` / `copyText`）、通知文案生成均复用，组件层面保持分离是"关注点分离"的正确边界，无需合并。

### 附：一个相关的隔离观察（不在本次范围，建议后续单独立项）

省地接社协作 H5 当前顶部仍展示"对客总价" chip（`data.guestPrice`）。从三角色隔离矩阵（见 `Account.vue`）看，省地接社的"对客价"权限为 ✗，该 chip 属于轻微数据泄漏。建议作为独立任务处理（隐藏/改为"由一手最终核定"占位），本次聚焦按钮合并，不擅自扩大改动面。

  - 通知文案框出现
  - 通过 `GET /h5/route/:token/feedback` 确认说明已作为 feedback 入库
