# 上线部署（选项 B：CloudBase 云托管 + 前端 CDN/EdgeOne）

> 配套前置：数据库与迁移见 `doc/05-数据库与部署.md`。本文聚焦"把链路打通到能上线"。
> 本地沙箱无 Docker / 无 CloudBase 凭证，因此**镜像构建与真实 CDN 上传需在你本地或 CI 中执行**；本文给出可直接运行的产物与命令。

---

## 1. 部署拓扑

```
                         ┌─────────────────────────────────────────────┐
   浏览器 / 微信          │           腾讯云 EdgeOne 加速域名            │
   (客户 / 旅行社 / 一手)  │  静态站点: 前端 SPA (frontend/dist)         │
        │                 │  SPA 回源重写: /* -> /index.html            │
        │   https://app.xxx.com                                         │
        ├────────────────►│  /api/* 反代(可选) ──┐                       │
        │                 └──────────────────────┼──────────────────────┘
        │                                        │ 内网/公网
        │                                        ▼
        │                         ┌──────────────────────────┐
        │                         │  CloudBase 云托管容器      │
        │  公开协作 H5 链接         │  backend (NestJS)        │
        │  /h5/route/:token  ─────►│  :3000  /api/*           │
        │                         │  健康检查: /api/health    │
        │                         └───────────┬──────────────┘
        │                                     │
        │                                     ▼
        │                         ┌──────────────────────────┐
        │                         │  CloudBase PostgreSQL     │
        │                         │  DATABASE_URL (sslmode)   │
        │                         └──────────────────────────┘
```

- **前端**：纯静态，部署到 COS + EdgeOne（或任意静态托管）。API 走同源 `/api`（由 EdgeOne 反代到云托管），或直接配 `VITE_API_BASE` 指向云托管域名（已开 CORS）。
- **后端**：CloudBase 云托管容器，监听 `PORT`（默认 3000），健康检查 `/api/health`。
- **数据库**：CloudBase PostgreSQL（见 `doc/05`）。

## 2. 后端 → CloudBase 云托管

### 2.1 准备

- 容器镜像由代码仓库构建，无需本地 Docker。
- 准备环境变量（**不要写进仓库**，在 CloudBase 云托管控制台配置）：
  - `DATABASE_URL`：CloudBase PostgreSQL 连接串，含 `sslmode=require`（见 `backend/.env.example`）
  - `JWT_SECRET`：随机长串（部署务必改）
  - `DEV_BYPASS_AUTH=false`（**生产必须关掉开发旁路**）
  - `WECHAT_APPID` / `WECHAT_SECRET`：填了走真实微信授权，留空则 `auth/wechat/callback` 回退演示账号
  - `PORT=3000`（云托管注入，通常无需手动设）

### 2.2 控制台步骤

1. 云托管 → 新建服务（例如 `pandaking9-backend`）。
2. 代码源：绑定本仓库（或推送到绑定的 Git 仓库）。
3. 构建配置：
   - **Dockerfile 路径**：`backend/Dockerfile`
   - **构建上下文目录**：`/`（仓库根，因为 `pnpm-workspace.yaml` 在根）
   - **监听端口**：`3000`
   - **健康检查路径**：`/api/health`
4. 环境变量：填入 2.1 中的变量。
5. 部署，等待镜像构建完成。

> 镜像构建用 `pnpm install --frozen-lockfile`，请确保 `pnpm-lock.yaml` 已提交（已提交）。
> 构建阶段会自动 `prisma generate` + `nest build`，运行阶段 `node backend/dist/main.js`。

### 2.3 数据库迁移与种子（一次性）

镜像部署后、首次运行前执行迁移（任选其一）：

- **本地执行**（推荐，使用 CloudBase PostgreSQL 的连接串）：
  ```bash
  cp backend/.env.example backend/.env   # 填入真实 DATABASE_URL 等
  pnpm -F backend prisma:deploy          # 应用 migrations/0_init
  pnpm -F backend db:seed                # 写入演示数据（一手/旅行社/省地接社/演示邀请）
  ```
- **云托管实例内执行**：通过 CloudBase 云托管「登录实例」或「执行命令」运行上面两条 `pnpm` 命令（需实例内有 pnpm，或用 `npx prisma`）。

## 3. 前端 → COS + EdgeOne

### 3.1 构建

```bash
# 默认 VITE_API_BASE=/api（配合 EdgeOne 反代）；如需直连后端，先改 frontend/.env.production
pnpm -F frontend build
# 产物：frontend/dist
```

如需 CDN 子路径部署：
```bash
VITE_BASE=/workbench/ pnpm -F frontend build
```

### 3.2 上传到 COS（EdgeOne 源站）

```bash
pnpm add -w cos-nodejs-sdk-v5
export COS_SECRET_ID=xxx COS_SECRET_KEY=xxx COS_BUCKET=xxx COS_REGION=ap-shanghai
node deploy/deploy-frontend.mjs
```

`deploy/deploy-frontend.mjs` 会：
- 校验 `frontend/dist` 存在；
- `index.html` 设 `no-cache`（发布即生效），带 hash 的静态资源设 `max-age=31536000, immutable`；
- 全量上传到指定 COS Bucket。

### 3.3 EdgeOne 控制台配置

1. 新建/接入加速域名（如 `app.xxx.com`），源站选择上面的 COS Bucket。
2. **SPA 回源重写**（关键）：对所有**非文件请求**（无扩展名）回源到 `/index.html`，保证 `/routes/list`、`/h5/route/:token` 等前端路由刷新不 404。
   - 注意：前端已改用 history 模式（无 `#`），**必须配置此重写**，否则深链接刷新会 404。
   - 协作 H5 分享卡片由**后端服务端渲染**：分享链接形如 `https://<后端域名>/share/route/:token`（带 OG 注入），不在静态托管，无需重写；`/share/og-cover.png` 为分享封面图。
3. **API 反代（推荐，可选）**：添加规则 `/api/*` 回源到 CloudBase 云托管后端域名。这样前端无需改 `VITE_API_BASE`，且规避跨域。
   - 若不做反代，则在 `frontend/.env.production` 设 `VITE_API_BASE=https://<cloudbase-domain>` 重新构建（后端已 `enableCors()`）。

## 4. 端到端验证清单

部署完成后逐项确认：

- [ ] `GET https://<cloudbase-domain>/api/health` 返回 `200 {"status":"ok"}`
- [ ] `GET https://<cdn-domain>/` 返回前端首页（SPA 壳）
- [ ] 访问 `/routes/kanban` 并刷新页面不 404（SPA 重写生效）
- [ ] `Account.vue` 三角色 `dev-login` 可登录（注：生产 `DEV_BYPASS_AUTH=false`，需走 `accept-invite` 或微信授权；本地联调用 dev 旁路）
- [ ] 新建路线 → 「保存并通知」→ 复制 H5 链接 → 匿名打开 `/h5/route/:token` 能看到脱敏行程与对客价
- [ ] H5 页提交反馈 → 后端 `RouteFeedback` 落库
- [ ] 数据库 `Route/CostInquiry` 等表已建（迁移成功）

## 5. 回滚与变量说明

- **回滚**：云托管支持历史版本，前端 COS 上传为覆盖式，建议在 `deploy` 脚本或 CI 中开启版本控制/保留上一版 `index.html`。
- **`DEV_BYPASS_AUTH`**：仅本地/联调为 `true`；生产必须 `false`，否则任何人都能以一手身份操作。
- **`JWT_SECRET`**：多实例部署必须保持一致，否则 token 互不认。
- **`DATABASE_URL`**：CloudBase PostgreSQL 默认要求 `sslmode=require`。
