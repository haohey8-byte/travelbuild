# 微信通知链接与三角色分享逻辑规划

## 一、问题根因（已取证）

### 1.1 截图中的错误链接
用户在微信中收到的链接形如：

```
https://travelbuild-d3gvgvtj70ddd0e43-1379227294.tcloudbaseapp.com/#/routes/a28cf956-2954-4b5e-96bc-4548b9eff82c
```

该链接指向 **前端控制台详情页**（`/routes/:id`），需要登录态，且在微信内置浏览器中无法直接打开已登录态，因此表现为「打不开 / 空白」。

### 1.2 链接从哪里生成
`frontend/src/views/RouteDetail.vue` 的 `onSubmitSuggestion`（境外旅行社 / 省地接社提交建议并通知一手）中，第 572–573 行直接拼接了控制台链接：

```ts
const base = (import.meta.env.VITE_BASE as string) || '/'
const link = `${window.location.origin}${base}#/routes/${id}`
```

这是错误的：通知文案应该发「公开免登录的 H5 协作页」，而不是需要登录的控制台详情页。

### 1.3 即使改用 H5 路径，域名仍然错误
项目里正确的 H5 分享页是后端 SSR 的 `/share/route/:token`（带 OG 标签，公开免登录）。但当前 `frontend/src/utils/share.ts` 的 `shareH5Url` 实现：

```ts
const API_BASE = (import.meta.env.VITE_API_BASE as string) || '/api'
const API_ORIGIN = API_BASE.replace(/\/api$/, '')
export function shareH5Url(token: string): string {
  return `${API_ORIGIN}/share/route/${token}`
}
```

当前生产环境未配置 `VITE_API_BASE`（默认 `/api`），导致 `API_ORIGIN = ''`，生成的是相对路径 `/share/route/{token}`。在微信中点击时，该相对路径会基于当前页面域名（前端静态托管域名）解析，而不是后端域名，因此仍然打不开。

## 二、三角色视角与链接策略

| 角色 | 操作场景 | 通知接收方 | 应生成的链接 | 可见数据 |
|---|---|---|---|---|
| **PandaKing（一手）** | 保存并通知客户/协作方 | 客户、境外旅行社微信群 | 后端 H5 分享页 `/share/route/:token` | 仅对客价（公开） |
| **Agency（境外旅行社）** | 提交报价建议并通知一手 | 一手 PandaKing 的微信群 | 后端 H5 分享页 `/share/route/:token` | 仅对客价（公开） |
| **Provincial（省地接社）** | 在 H5 页提交反馈 / 编辑行程 | 一手 PandaKing | 当前 H5 页链接或 `/share/route/:token` | 仅自身成本① |

核心原则：
- **所有对外通知文案都必须使用公开免登录的 H5 分享页**，确保微信内点击可打开。
- **H5 分享页只暴露对客价（guestPrice）**，不暴露省地接社成本、PandaKing 利润等内部数据。
- **控制台详情页（`/routes/:id`）仅用于登录后的工作台内部协作**，不应出现在任何微信通知文案中。

## 三、修复方案

### 3.1 前端环境变量：明确后端域名
新增 `frontend/.env.production`：

```ini
# 生产环境：后端 CloudBase 云托管域名（用于构造 H5 分享链接，必须是绝对 URL）
VITE_API_BASE=https://travelbuild-281496-9-1379227294.sh.run.tcloudbase.com/api
```

这样 `shareH5Url` 可解析出 `API_ORIGIN = https://travelbuild-281496-9-1379227294.sh.run.tcloudbase.com`，生成绝对 URL 的分享链接。

### 3.2 修改 shareH5Url 的解析逻辑
增强容错：
- 若 `VITE_API_BASE` 是绝对 URL，去掉 `/api` 得到后端 origin。
- 若 `VITE_API_BASE` 是相对路径（如 `/api`），则回退到 `/share/route/{token}`（仅用于开发期走 Vite 代理）。

### 3.3 修改 RouteDetail.vue 的 onSubmitSuggestion
不再拼接 `#/routes/${id}`，改为：
1. 调用 `shareRoute(id, role)` 获取 share token。
2. 用 `shareH5Url(token)` 生成 H5 分享链接。
3. 将链接写入通知文案。

这样既保证链接公开可打开，又保持与「保存并通知」一致的 H5 分享机制。

### 3.4 数据隔离
- `shareRoute` 调用时传 `role: 'agency'`（或默认 public=true），后端 `createShare` 会按 `serializeVersion(..., role)` 做字段隔离。
- H5 分享页最终由 `og-page.controller.ts` 渲染，使用 `maskQuotePublic` 进一步脱敏，只显示对客价。

## 四、验证计划

1. **本地构建**：`npm run build` 无错误。
2. **部署**：`bash deploy/deploy.sh` 上传到 CloudBase 静态托管。
3. **线上实跑**：用系统 Chrome + Playwright 模拟微信场景：
   - 用 Agency 角色登录，进入 Jely澳洲 详情页。
   - 点击「提交建议并通知一手」。
   - 采集生成的通知文案中的链接，检查是否为 `https://travelbuild-281496-9-1379227294.sh.run.tcloudbase.com/share/route/{token}`。
   - 直接访问该链接，确认页面正常渲染（200、有行程、有对客价、无控制台导航）。
   - 检查 console 无错误。

## 五、风险与回退

- 修改 `VITE_API_BASE` 为绝对 URL 后，前端 API 请求将直接走后端域名（依赖后端 CORS）。后端已 `enableCors()`，风险可控。
- 若 CloudBase 静态托管的 `/api/*` 反代仍然生效，相对路径 `/api` 其实也会工作；但绝对 URL 更明确，且不影响分享链接。
- 若未来切换域名，只需改 `.env.production` 中的后端域名。
