# 页面设计：客户看板（Customer Kanban）

> 状态：设计稿（待开发）  
> 目标：参照线框图重新组织「路线管理」首页，使其从「路线卡片」转向「客户视角」的看板，强化销售漏斗、待办提醒与快速筛选。  
> 关联页面：保留「归档历史」（`/route-archives`）和「路线详情」（`/routes/:id`），仅重设计本页入口。

---

## 1. 设计目标

1. **客户视角优先**：把「路线」包装成「客户行程」，首屏展示客户名、国籍、出行日期、人数，而不是内部版本号。
2. **销售漏斗可视化**：顶部用四张关键数字卡（活跃客户 / 待报价 / 待跟进 / 本月成单），让一手一眼看到当前工作负荷。
3. **多维度筛选**：支持按状态、旅行社、目的地、客户名/目的地搜索，避免当前只能按状态 chip 过滤的限制。
4. **待办提醒落地**：把「需要人处理」的事集中展示在底部，减少遗漏。
5. **文案统一**：将右上角按钮从「+ 创建路线」改为「+ 新建路线」；保留「归档历史」入口。

## 3. 区域详细设计

### 3.1 页面标题与全局操作

| 元素 | 内容 | 交互 |
|------|------|------|
| 页面标题 | 客户看板 | 读 |
| 副标题 | 管理所有境外游客的定制行程与报价 | 读 |
| 归档历史 | 按钮 `归档历史` | 仅一手可见，跳转 `/route-archives` |
| 新建路线 | 按钮 `+ 新建路线` | 打开新建路线抽屉/弹窗（原「创建路线」弹窗） |

> **文案变更**：原「+ 创建路线」→「+ 新建路线」。

### 3.2 统计卡片区

四张卡片，按从左到右排列：

| 指标 | 显示名 | 计算规则（基于当前用户可见路线） | 颜色/数字 |
|------|--------|----------------------------------|-----------|
| 活跃客户 | 活跃客户 | 所有 `statusKey !== 'lost'` 的路线数 | 数字黑色 |
| 待报价 | 待报价 | `statusKey === 'awaiting_quote'` 的路线数 | 数字琥珀色 |
| 待跟进 | 待跟进 | `statusKey === 'pending_followup'` 的路线数 | 数字红色 |
| 本月成单 | 本月成单 | `statusKey === 'booked'` 且 `route.updatedAt` 在本月的路线数 | 数字绿色 |

卡片样式：圆角白底卡片、轻微阴影、上边缘 4px 品牌色线、标题小号灰色、数字 28px 加粗。

> **数据说明**：当前 `Route` 模型有 `updatedAt`，无需新增字段；但列表接口需要返回 `updatedAt` 供前端统计。

### 3.3 筛选栏

从左到右：

| 控件 | 类型 | 默认值 | 数据来源 | 交互 |
|------|------|--------|----------|------|
| 状态 | 下拉选择 | 全部状态 | 现有 `RouteStatusKey` 映射 | 选择后联动表格 |
| 旅行社 | 下拉选择 | 全部旅行社 | 当前用户可见的机构列表（`fetchAgencies`） | 选择后联动表格 |
| 目的地 | 下拉选择 | 全部目的地 | 从当前路线 `destination` 去重生成 | 选择后联动表格 |
| 搜索 | 输入框 | 空 | 占位文案「搜索客户名/目的地…」 | 回车或点击搜索按钮触发 |
| 搜索按钮 | 按钮 | - | - | 触发搜索，重置分页到第 1 页 |

> **后端依赖**：`GET /routes` 需扩展参数：`status`、`agencyId`、`destination`、`q`（搜索关键字）。当前仅支持 `status`。

### 3.4 客户列表表格

表格列（与线框图对应）：

| 列 | 显示内容 | 数据来源 |
|----|---------|----------|
| 客户 | 圆形头像（客户名首字母）+ 客户名 + 副标题（国家 · 人数 · 出发日期） | `customerNameCn` / `customerName`、`country`、`groupSize`、`travelDate` |
| 旅行社 | 旅行社名称 | `agency` |
| 目的地 | 目的地名 + 第二行具体景点（如「九寨沟+毕棚沟」） | `destination`；当前模型无细分景点，第二行可省略或后续扩展 |
| 行程模式 | 标签 `自行` / `协作` | `modeKey === 'solo' ? '自行' : '协作'` |
| 状态 | 彩色标签 | `statusKey` 映射 |

行交互：点击整行进入 `/routes/:id`。

头像规则：取客户名（优先 `customerNameCn`，无则 `customerName`）首字母，大写，背景色按客户名哈希分配。

状态标签样式（与现有 `RouteStatusKey` 对应）：

| statusKey | 显示文案 | 标签颜色 |
|-----------|---------|----------|
| consulting | 咨询中 | 灰色 |
| awaiting_pk_confirm | 待一手确认 | 橙色 |
| awaiting_agency_revision | 待旅行社修订 | 橙色 |
| awaiting_quote | 待报价 | 蓝色 |
| awaiting_feedback | 待反馈 | 紫色 |
| awaiting_confirm | 待确认 | 青色 |
| confirmed | 已确认 | 绿色 |
| booked | 已成单 | 深绿色 |
| pending_followup | 待跟进 | 红色 |
| lost | 已流失 | 深灰（表格中一般不显示，因默认过滤掉） |

### 3.5 待办提醒区

独立卡片区域，标题「待办提醒」。

| 元素 | 说明 |
|------|------|
| 列表项 | 每行一条待办：左侧彩色方块标记类型，中间描述，右侧操作按钮 |
| 类型标记 | 红色=报价/行程超期；紫色=审核；黄色=等待他人回复；蓝色=快照过期 |
| 操作按钮 | 根据待办类型显示不同文案：跟进提醒 / 催办 / 去审核 / 续期 |

待办类型（V1 可基于现有数据生成，无需新增后端）：

| 待办内容示例 | 触发条件 | 操作按钮 | 点击后行为 |
|-------------|----------|----------|------------|
| 报价已发出 X 天，旅行社未回复 | 最新版本非草稿且 `awaiting_feedback` | 跟进提醒 | 跳转详情 → 客户与版本 Tab |
| 行程已分配给省地接社，等待成本价 | 存在 `pending` 的 CostInquiry | 催办 | 跳转详情 → 复制协作链接 |
| 新微信授权账号待审核 | 当前无需后端，可预留 | 去审核 | 跳转账号管理 |
| 报价快照将于明天过期 | 当前无需后端，可预留 | 续期 | 弹出续期确认 |

> **V1 实现建议**：先实现前两类（基于 RouteVersion + CostInquiry），后两类作为占位预留，避免一次性引入审核/过期机制。

## 4. 数据需求与接口变更

### 4.1 前端类型扩展（可选）

`Route` 接口当前已包含所需字段，无需修改；但列表接口需要返回 `updatedAt` 用于「本月成单」统计：

```ts
// 可新增或确认已有
export interface Route {
  // ... 已有字段
  updatedAt?: string // 用于本月成单统计
}
```

### 4.2 API 参数扩展

`GET /routes` 当前参数：

```ts
{ status?: string; role?: string }
```

扩展为：

```ts
{
  status?: string;        // 状态筛选
  agencyId?: string;      // 旅行社筛选
  destination?: string;   // 目的地筛选
  q?: string;             // 客户名/目的地模糊搜索
}
```

### 4.3 后端查询逻辑

- `status`：精确匹配 `Route.statusKey`。
- `agencyId`：精确匹配 `Route.agencyId`；若当前角色为旅行社，后端仍按原有隔离只返回本机构。
- `destination`：精确匹配 `Route.destination`（前端下拉选项从已有数据去重）。
- `q`：对 `customerName`、`customerNameCn`、`destination` 做 `ILIKE` 模糊匹配，三字段任一命中即可。

### 4.4 统计数字

由前端在已请求列表内计算，减少一次额外请求。若未来路线数量大，可再增加 `GET /routes/stats` 聚合接口。

## 5. 交互流程

1. 进入页面：请求 `GET /routes?status=all`（默认全部），加载统计卡、筛选下拉选项、表格、待办。
2. 筛选：任一筛选条件变化 → 重新请求 `/routes` 并替换表格，同时更新统计数字。
3. 搜索：输入关键词 → 点击「搜索」或回车 → 触发 `q` 参数搜索。
4. 新建路线：点击「+ 新建路线」→ 打开弹窗（沿用现有新建逻辑）→ 创建成功后刷新列表。
5. 行点击：进入 `/routes/:id`。
6. 待办操作：点击按钮 → 跳转到对应详情页或触发对应动作。

## 6. 保留与废弃

### 6.1 保留

- 「归档历史」按钮与页面 `/route-archives` 继续存在。
- 新建路线弹窗字段（客户名、中文名、目的地、国家、旅行社、人数、出行日期、模式）保持不变，仅按钮文案改为「新建路线」。
- 路线详情页 `/routes/:id` 不动。

### 6.2 调整

- 当前看板卡片布局替换为表格布局。
- 当前顶部状态 chip 过滤替换为「状态 + 旅行社 + 目的地」组合筛选。
- 顶部统计从「全状态计数」改为四张关键业务卡。
- 文案：「创建路线」→「新建路线」。

### 6.3 不改动

- 路线状态机（`RouteStatusKey`）不改动。
- 机构隔离逻辑（`agencyId` / `provincialId`）不改动。
- 权限矩阵（仅一手可见归档历史）不改动。

| awaiting_pk_confirm | 待一手确认 | c-amber | #FAC775 | #633806 |

| awaiting_feedback | 待反馈 | c-purple | #AFA9EC | #534AB7 |

### 10.2 行程模式 / 统计卡 / 待办

- 行程模式：`自行`=c-blue，`协作`=c-purple（浅填充+深色字胶囊）

- 主操作按钮（新建路线）固定品牌色 c-teal 600 `#0F6E56`
- 页面底色 `#F4F5F7`，卡片白底 + `0.5px` 边框，扁平无阴影

### 10.3 头像/首字母底色

按客户名哈希分配 9 色阶 200 档（蓝/琥珀/绿/粉/青等），白色首字母。
