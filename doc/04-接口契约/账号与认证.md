# 接口契约：账号与认证

## 角色与拓扑
- **PandaKing（一手）**：衔接上游「境外旅行社」与下游「省地接社」的枢纽，可见全部数据。
- **境外旅行社 / 省地接社**：彼此**完全物理隔绝**——无论价格还是信息，互不知道对方的存在与数据展示（见《权限矩阵》）。
- 每个机构有 **管理员(admin) / 员工(staff)** 两级。

## 两层级邀请模型（邀请制，无公开注册）
1. **一手 PandaKing** 通过「复制 H5 邀请链接」邀请境外旅行社 / 省地接社的**管理员**（role + agencyId + level=admin）。
2. 受邀管理员（或一手）再邀请**本机构员工**（level=staff，agencyId 强制继承邀请人机构）。
3. 受邀者打开 H5 链接 `/h5/invite/:token` → 填名称 → 接受邀请 → 账号绑定该机构与层级。

权限约束（服务端强制）：
- 仅一手可创建「机构管理员」邀请。
- 仅一手或「本机构管理员」可创建「机构员工」邀请（staff 邀请的 agencyId 必须等于邀请人机构，物理隔绝）。

## 鉴权流程
1. 微信网页授权（OAuth2）：`GET /api/auth/wechat?redirect=...` → 微信回调 `code` → 换 openid → 匹配/创建用户 → 发 JWT。
2. 邀请制：无公开注册。一手（或管理员）发邀请 → 受邀者经 H5 链接接受 → 绑定角色+机构+层级。

## 端点
| Method | Path | 说明 | 权限 |
|---|---|---|---|
| GET | `/api/auth/wechat` | 发起微信 OAuth2 | 公开 |
| POST | `/api/auth/wechat/callback` | code 换 token | 公开 |
| POST | `/api/invites` | 创建邀请（role + agencyId + level + email?） | 一手 / 本机构管理员 |
| GET | `/api/invites` | 邀请列表（按机构隔离） | 登录 |
| GET | `/api/invites/:token` | 邀请详情（角色/机构预览） | 持 token |
| POST | `/api/accept-invite` | 接受邀请 → 建账号 + 发 JWT（继承 agencyId/level/parentId） | 持 token |
| GET | `/api/members` | 成员列表（**按机构物理隔绝**：一手见全部，机构仅见本机构） | 登录 |
| PUT | `/api/members/:id/role` | 改角色 | 一手 |
| POST | `/api/members/:id/disable` | 停用（失效 token） | 一手 |
| GET | `/api/me` | 当前用户信息+角色+机构+层级 | 登录 |

## JWT
- 载荷：userId, role, **agencyId, level**, name。
- 过期：access 2h，refresh 7d。
- 停用后 refresh 黑名单。
- 服务端所有隔离判断（路线列表、成员列表、字段可见性）均以 JWT 中的 role/agencyId 为准，不依赖前端隐藏。
